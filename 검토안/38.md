있다. 다만 이제는 “핵심 누락”보다는 “완성도 상향” 영역이다.

추가로 넣을 만한 마지막 항목은 아래다.

# 1. 학습 데이터 오염 방지

```text
paper 결과와 live 결과를 섞어 학습하지 않는다.
shadow 후보끼리도 성과 집계를 분리한다.
비정상 데이터 구간은 학습 샘플에서 제외한다.
```

# 2. 성과 착시 방지

```text
수익률만 보지 말고 exposure time 대비 수익을 본다.
소수 대박 거래가 전체 성과를 왜곡하는지 확인한다.
거래 수가 부족한 전략은 승인하지 않는다.
```

# 3. 시장 충격 회피

```text
거래량 대비 주문 크기 제한
호가 잔량 대비 주문 크기 제한
얇은 호가에서 시장가 금지
```

# 4. 시스템 자기진단

```text
engine health
data health
execution health
risk health
ledger health
dashboard health
```

각 health가 나쁘면 신규 진입 차단.

# 5. 운영 모드별 권한 분리

```text
REPLAY: 실험 가능
PAPER: 가상체결만 가능
SHADOW: 실시간 감시, 실제 주문 금지
MICRO_LIVE: 최소 주문만 가능
LIVE: 승인 snapshot만 가능
```

# 6. 실수 방지 UX

```text
LIVE 실행 전 현재 승인 snapshot 표시
현재 risk cap 표시
현재 API 권한 상태 표시
LIVE 불가 사유 표시
STOP은 항상 즉시 가능
```

# 7. 성과 악화 시 자동 롤백

```text
새 snapshot 적용 후 성과 악화 시 이전 승인 snapshot으로 복귀
롤백 기록 보존
롤백 후 일정 시간 risk-down
```

# 8. 외부 장애 대응

```text
인터넷 불안정
DNS 장애
거래소 응답 지연
로컬 디스크 부족
시스템 시간 틀어짐
```

감지 시 no-trade 또는 kill switch.

# 9. 포지션 없는 상태와 있는 상태의 로직 분리

```text
flat state: 신규 진입 평가
position state: 추가 진입/청산/축소만 평가
uncertain state: 신규 진입 금지
```

# 10. 최종 원칙

```text
수익률 목표보다 생존성이 우선이다.
애매하면 거래하지 않는다.
검증되지 않은 것은 live에 연결하지 않는다.
데이터가 의심되면 신호도 무효다.
```

이 정도까지 넣으면 설계도와 지침서 기준으로는 충분히 과하다 싶을 만큼 촘촘하다. 이후부터는 문서 추가보다 구현, 검증, 페이퍼 데이터 축적이 더 중요하다.
