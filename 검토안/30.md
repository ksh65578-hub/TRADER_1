구현 목표는 “개념 반영”이 아니라 아래 수준까지 닫혀야 한다.

# 기준

설계도/지침서에 이미 있는 내용은 방향성이다. 구현에서는 다음 4개가 모두 있어야 한다.

| 단계  | 의미                    |
| --- | --------------------- |
| 선언  | 문서에 개념이 있음            |
| 계약화 | 입력/출력/schema/상태값이 고정됨 |
| 실행화 | 런타임 코드가 실제로 계산/판단/차단함 |
| 검증화 | 테스트와 산출물로 통과 여부를 증명함  |

지금 필요한 건 최소 3단계, 가능하면 4단계다.

# 1. Exchange / market_type 구현

문서상 목표:
Upbit + Binance spot + Binance USD-M futures.

구현 필요 수준:

| 항목              | 필요한 구현                                                                                                               |
| --------------- | -------------------------------------------------------------------------------------------------------------------- |
| Upbit spot      | adapter, symbol filter, tick size, fee, min order, balance/position abstraction                                      |
| Binance spot    | adapter, symbol filter, fee, lot size, min notional, order status mapping                                            |
| Binance futures | USD-M futures adapter, 1x leverage enforcement, one-way/hedge mode 명확화, long/short position abstraction              |
| namespace       | exchange / market_type / mode 별 데이터 완전 분리                                                                            |
| launcher        | UPBIT_PAPER, UPBIT_LIVE, BINANCE_SPOT_PAPER, BINANCE_SPOT_LIVE, BINANCE_FUTURES_PAPER, BINANCE_FUTURES_LIVE 수준으로 명확화 |

최소 통과 기준:

```text
exchange=BINANCE, market_type=FUTURES 일 때
short 후보 생성, paper 체결, ledger 기록, dashboard 표시까지 된다.
```

# 2. 실제 paper/shadow trading loop

현재처럼 report 생성 후 종료하면 안 된다.

필요한 루프:

```text
market data ingest
→ canonical event
→ feature update
→ regime classify
→ strategy signal
→ risk/sizing
→ paper/shadow execution
→ ledger/reconciliation
→ metrics
→ dashboard/readiness update
```

필수 조건:

| 항목               | 필요한 구현                                  |
| ---------------- | --------------------------------------- |
| paper engine     | 실제 가상 잔고, 포지션, 체결, 수수료, 슬리피지 반영         |
| shadow engine    | 여러 파라미터 후보 병렬 비교                        |
| live engine      | 기본 차단, LIVE_READY snapshot 없으면 절대 주문 금지 |
| restart recovery | 재시작해도 잔고/포지션/ledger 복구                  |
| idempotency      | 같은 event/order를 중복 반영하지 않음              |

최소 통과 기준:

```text
PAPER 모드가 지속 실행되고,
가상 주문/체결/잔고/손익/포지션이 시간에 따라 변해야 한다.
```

# 3. Regime router

문서에 “상승/횡보/하락/패닉/저변동”이 있어도 부족하다.

구현 필요 수준:

| regime    | 판단 기준 예시                                            | 허용 전략                     |
| --------- | --------------------------------------------------- | ------------------------- |
| uptrend   | EMA50 > EMA200, slope > threshold, ADX >= threshold | pullback, breakout        |
| range     | ADX 낮음, price near VWAP band                        | VWAP mean reversion       |
| downtrend | EMA50 < EMA200, slope < threshold                   | no-trade 또는 futures short |
| panic     | vol spike, spread spike, liquidity drop             | no-trade                  |
| quiet     | low realized vol, low volume                        | no-trade                  |

필수 구현:

```text
regime_score
regime_confidence
allowed_strategy_set
blocked_reason
```

최소 통과 기준:

```text
하락장으로 판정되면 Upbit spot long 진입이 불가능해야 한다.
Binance futures에서는 조건 충족 시 short 후보가 생성되어야 한다.
```

# 4. 전략 엔진

최소 3개는 실제로 작동해야 한다.

| 전략                  | 구현 필요                                                   |
| ------------------- | ------------------------------------------------------- |
| pullback trend      | 추세 중 눌림, 재상승 확인, 손절/익절                                  |
| breakout            | range high / volatility expansion / volume confirmation |
| VWAP mean reversion | VWAP band 이탈 후 복귀, range regime 한정                      |
| futures short       | downtrend 또는 breakdown 조건에서 short                       |
| no-trade engine     | 나쁜 구간에서 강제 비활성                                          |

각 전략은 반드시 이 값을 출력해야 한다.

```text
signal_side
entry_reason
no_entry_reason
confidence
expected_edge_after_cost
stop_loss
take_profit
time_stop
invalidated_when
```

최소 통과 기준:

```text
진입하지 않을 때도 “왜 안 들어갔는지”가 dashboard와 log에 남아야 한다.
```

# 5. 비용 포함 기대값 계산

월 +5% 이상을 노리려면 이게 핵심이다.

구현 필요:

```text
gross_expected_move
fee_cost
spread_cost
slippage_cost
funding_cost
net_expected_edge
```

진입 조건:

```text
net_expected_edge > minimum_required_edge
```

거래소별로 수수료/틱/최소주문/수량단위가 다르게 적용되어야 한다.

최소 통과 기준:

```text
비용 차감 후 기대값이 음수면 signal이 강해도 진입 금지.
```

# 6. Position sizing

문서에 “확신도 기반”만 있으면 부족하다.

구현 필요:

```text
risk_per_trade
confidence_multiplier
regime_multiplier
drawdown_multiplier
liquidity_multiplier
max_position_cap
daily_loss_cap
```

예시 구조:

```text
base_risk = equity * 0.0025
if strong_signal: risk = base_risk * 2
if drawdown_active: risk = base_risk * 0.5
if panic_or_bad_liquidity: risk = 0
```

최소 통과 기준:

```text
같은 전략이라도 confidence와 drawdown 상태에 따라 주문 크기가 달라져야 한다.
```

# 7. Risk engine

필수 차단 조건:

```text
daily_loss_limit
weekly_loss_limit
monthly_loss_limit
max_drawdown_limit
max_open_positions
max_symbol_exposure
max_exchange_exposure
kill_switch
stale_data_block
spread_block
slippage_block
reconciliation_mismatch_block
```

최소 통과 기준:

```text
daily loss limit 도달 시 모든 신규 진입이 자동 차단되어야 한다.
```

# 8. LIVE_READY snapshot

가장 중요하다.

LIVE는 “현재 파라미터”를 직접 쓰면 안 된다. 반드시 승인된 snapshot만 써야 한다.

필요한 snapshot 내용:

```text
exchange
market_type
strategy_id
symbol_universe
parameter_set
paper_period
shadow_period
trade_count
profit_factor
expectancy
max_drawdown
win_rate
avg_win
avg_loss
fee_slippage_model
regime_breakdown
approved_at
hash
```

LIVE_READY 조건 예시:

```text
paper/shadow 30~60일 이상
trade_count 100~300회 이상
profit_factor >= 1.25
net_expectancy > 0
max_drawdown <= allowed_limit
fee/slippage 반영 후 양수
regime별 손실 폭 제한
reconciliation mismatch 없음
```

최소 통과 기준:

```text
LIVE_READY snapshot 없이는 LIVE 실행기를 켜도 주문이 절대 나가지 않아야 한다.
```

# 9. Dashboard 구현

사용자 목표는 단순해야 한다.

대시보드는 최소 이걸 보여야 한다.

```text
현재 모드
LIVE_READY 여부
live_order_allowed 여부
equity
cash
positions
오늘 손익
월 손익
drawdown
active regime
active strategy
candidate symbols
entry reasons
no-entry reasons
risk blocks
data health
exchange connection health
```

최소 통과 기준:

```text
사용자는 “켜도 되는지 / 왜 안 되는지 / 돈이 어떻게 변하는지”를 한 화면에서 알아야 한다.
```

# 10. 검증/테스트

구현은 테스트 없으면 닫힌 게 아니다.

필수 테스트:

```text
live blocked by default
no live without snapshot
downtrend blocks spot long
futures short paper fill
fee/slippage negative edge blocks entry
daily loss limit blocks entry
duplicate event not double-counted
restart restores ledger/positions
exchange namespace separation
paper loop produces changing equity
dashboard emits LIVE_READY=false initially
```

최소 통과 기준:

```text
pytest가 timeout 없이 통과해야 한다.
PAPER 실행이 멈추지 않고 정상 heartbeat를 갱신해야 한다.
```

# 최종 기준

너의 설계도와 지침서에 더해서 구현은 최소 여기까지 가야 한다.

```text
1. 거래소/마켓 타입별 실제 adapter
2. 지속 실행되는 paper/shadow loop
3. 레짐 판정 공식과 전략 라우팅
4. 실제 전략 signal 생성
5. 비용 차감 기대값 계산
6. confidence 기반 position sizing
7. 손실 제한 risk engine
8. LIVE_READY snapshot 승인 구조
9. dashboard에서 상태/이유 표시
10. pytest와 acceptance artifact로 검증
```

결론:

```text
지금 필요한 구현 수준은
“설계 개념을 코드에 옮기는 것”이 아니라
“AI가 임의 판단하지 않아도 실행, 차단, 승인, 복구가 모두 결정되는 상태”다.
```


월 10%는 목표로 둘 수는 있다. 다만 구조가 달라진다.

월 10%는 일평균 약 0.32% 복리다. 자동매매 기준으로는 상당히 공격적인 목표다.

필요 조건은 이렇다.

| 영역     |         월 5% 구조 |             월 10% 구조 |
| ------ | --------------: | -------------------: |
| 거래 빈도  |              중간 |                 더 높음 |
| 자본 배분  |       차등 sizing |      강한 신호 집중 sizing |
| 하락장 대응 | 회피 또는 제한적 short |  futures short 적극 활용 |
| 전략 수   |            3~4개 |            레짐별 다중 엔진 |
| 손실 한도  |             보수적 | 손실 한도는 유지하되 회복 엔진 필요 |
| 검증 기준  |          30~60일 |            60~90일 이상 |
| 위험     |              중간 |                   높음 |

핵심은 단순히 리스크를 키우는 게 아니다.

```text
월 10% = 더 큰 베팅이 아니라
좋은 구간에 더 많은 자본을 배치하고
나쁜 구간을 더 빨리 끊는 구조
```

필수 보강은 아래다.

```text
1. Binance futures 1x long/short 완전 구현
2. 하락장 short 엔진 필수화
3. breakout 엔진을 월 수익률 주력 엔진으로 승격
4. shadow 후보를 여러 개 병렬 운용
5. 성과 좋은 전략/파라미터로 자동 자본 이동
6. 약한 신호는 거래 금지
7. 강한 신호에만 집중 sizing
8. 손실일 이후 자동 risk-down
9. 연속 손실 후 no-trade cooling
10. 월간 LIVE_READY 기준을 전략별로 분리
```

월 10% 목표용 sizing은 대략 이런 방향이다.

```text
weak signal: no trade
normal signal: 0.25R ~ 0.5R
strong signal: 1.0R
exceptional signal: 1.5R ~ 2.0R
```

대신 손실 제한은 더 강해야 한다.

```text
daily loss cap: -1.0% ~ -1.5%
weekly loss cap: -3% ~ -4%
monthly hard stop: -8% ~ -10%
consecutive loss cooling: required
```

LIVE_READY 기준도 더 높아야 한다.

```text
paper/shadow: 60~90일 이상
trade count: 300회 이상
profit factor: 1.35~1.50 이상
net expectancy: 비용 차감 후 양수
max drawdown: 월 목표수익률 이하 또는 1.2배 이내
regime별 손실 구간 확인
out-of-sample 성과 확인
```

결론:

```text
월 10%를 목표로 하려면
현재 설계 방향은 유지하되
Binance futures short, breakout 강화, dynamic sizing, shadow 경쟁 구조,
LIVE_READY 고강도 검증을 필수 구현으로 올려야 한다.
```

가능은 하지만, 월 5%보다 실패 확률과 변동성이 크게 올라간다.

맞다.

정확히는 이렇게다.

```text
기존 설계도/지침서의 방향성은 유지
+
수익성 개선 루프를 더 구체화
+
월 10% 목표용 필수 구현 조건을 추가
```

추가해야 할 핵심은 7개다.

```text
1. 월 10%는 목표값이지 보장값이 아니라는 명시

2. paper/shadow 단계에서 전략·파라미터 후보를 병렬 경쟁시킴

3. 성과 좋은 전략/파라미터에 자동으로 자본 배분을 늘림

4. Binance futures 1x long/short를 월 수익률 개선용 필수 경로로 명시

5. weak signal은 거래하지 않고, strong/exceptional signal에만 집중 sizing 적용

6. 손실일·연속 손실·drawdown 발생 시 자동 risk-down/cooling/no-trade

7. LIVE_READY snapshot은 월 10% 목표 기준으로 더 엄격하게 승인
```

즉 지금 문서에 이미 있는 구조는:

```text
레짐 판단
전략 선택
paper/shadow
자동 개선
LIVE_READY
fail-closed
```

추가로 닫아야 하는 구조는:

```text
월 10% 목표를 위한 성과 경쟁, 자본 배분, short 활용, sizing, risk-down, LIVE_READY 기준
```

한 줄 결론:

```text
설계 철학을 바꾸는 게 아니라,
이미 넣어둔 수익성 개선 루프를 월 10% 목표 수준으로 더 구체화해야 한다.
```
