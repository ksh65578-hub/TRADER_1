7차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
테스트, 실행 재현성, 패키징, source bundle, 산출물 위생

핵심 추가 발견

1. 패키지에 캐시 산출물이 대량 포함됨

확인 수량:

* 전체 파일: 1,806개
* .pyc 파일: 340개
* **pycache** 디렉터리: 48개

문제:
소스 패키지에 런타임 캐시가 포함되어 있습니다. 재현성, diff, 해시, 배포 위생 측면에서 명백한 결함입니다.

2. 대형 단일 파일이 과도함

대표 대형 파일:

* TRADER_1.md: 약 1.16MB
* trader1/validation/mvp0_validators.py: 약 453KB
* trader1/dashboard/read_only_dashboard.py: 약 355KB
* contracts/security/source_bundle_manifest.json: 약 298KB
* AGENTS.md: 약 292KB
* tests/dashboard/test_read_only_dashboard.py: 약 117KB

문제:
단일 파일이 너무 커서 import, 테스트, 리뷰, AI 패치, 회귀 추적이 모두 느려집니다.

특히 mvp0_validators.py는 validator registry/runner/개별 validator가 한 파일에 과밀하게 들어간 상태로 보입니다.

3. Python 프로젝트 메타데이터가 없음

확인되지 않음:

* pyproject.toml
* requirements.txt
* pytest.ini
* tox.ini
* setup.cfg
* Makefile
* .github workflow

문제:
재현 가능한 실행 환경이 닫혀 있지 않습니다.

결과:

* 어떤 Python 버전이 기준인지 불명확
* pytest 옵션 불명확
* 의존성 설치 기준 없음
* CI 기준 없음
* 최소 통과 명령어 없음

4. source_bundle_manifest가 루트 실행기를 포함하지 않음

contracts/security/source_bundle_manifest.json 기준 included_files에 다음이 없습니다.

* UPBIT_PAPER.py
* UPBIT_LIVE.py
* BINANCE_PAPER.py
* BINANCE_LIVE.py

문제:
루트 실행기가 실제 사용자 실행 표면인데 source bundle manifest에 포함되지 않습니다.

이는 release/source identity 검증에서 치명적입니다. 실행기가 바뀌어도 source bundle 검증이 못 잡을 수 있습니다.

5. source bundle hygiene가 “후보 번들”만 깨끗하게 보는 구조

manifest는 다음처럼 나옵니다.

* forbidden_count: 0
* included_files 내 .pyc: 0
* excluded_files 내 .pyc: 340

즉, 패키지에 .pyc가 있어도 “candidate bundle에서는 제외했으니 PASS”에 가까운 구조입니다.

문제:
사용자가 받는 실제 zip 자체가 오염되어 있는데 validator는 실패하지 않습니다.

필요한 validator:

* source_candidate_bundle_hygiene
* shipped_package_hygiene

둘을 분리해야 합니다.

6. 테스트 실행성이 불량함

단일 테스트 실행도 제한 시간 내 종료되지 않는 케이스가 있습니다.

확인:

* tests/contract/test_schema_instance_validation.py 실행이 제한 시간 내 종료 실패
* build_launcher_report 직접 호출도 일반 python 환경에서 종료성 문제가 관찰됨

주의:
python -S 환경에서는 일부 단순 확인이 가능했습니다. 즉 site-packages, pytest 환경, import side effect, 또는 런처 모듈 종료 처리와 충돌 가능성이 있습니다.

판정:
현재 테스트는 “빠르게 반복 가능한 회귀 안전망”이 아닙니다.

7. tests가 많지만 성격이 섞여 있음

테스트 파일은 많습니다.

하지만 현재 구조는 다음이 섞여 있습니다.

* 단위 테스트
* schema 테스트
* evidence 생성 검증
* dashboard 대형 fixture 검증
* 런처 실행 검증
* patch_result 검증
* validator registry 검증

문제:
빠른 smoke / contract / integration / slow 테스트 구분이 없습니다.

필요:

* tests/smoke
* tests/unit
* tests/contract
* tests/integration
* tests/slow
* tests/release

그리고 기본 pytest는 smoke + unit + core contract만 30초 내 통과해야 합니다.

8. tools 디렉터리가 evidence emitter로 과밀함

tools에 emit_*_patch_evidence.py가 대량 존재합니다.

문제:
패치마다 별도 emitter가 누적되어 유지보수성이 낮습니다.

권장:

* 공통 evidence writer 1개
* patch metadata input
* template 기반 emit
* 개별 emitter 스크립트는 제거 또는 generated로 이동

9. generated/runtime/evidence 경계가 혼란스러움

대량의 system/evidence, patch_results, validator_runs, runtime 산출물이 패키지에 포함되어 있습니다.

문제:
소스 패키지인지, 검증 산출물 포함 패키지인지 경계가 흐립니다.

권장 패키지 분리:

* source package
* verification evidence package
* runtime artifact package
* release package

현재는 이 경계가 섞여 있습니다.

결론

현재 패키지는 구현 스캐폴드보다 “누적 산출물까지 함께 들어간 작업 디렉터리 압축본”에 가깝습니다.

실제 구현을 계속 진행하려면 먼저 패키지 위생과 테스트 재현성을 고쳐야 합니다. 그렇지 않으면 이후 패치가 맞게 들어갔는지 검증하기 어렵습니다.
