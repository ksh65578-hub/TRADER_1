8차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
보안, 시크릿, API key 경계, 로그/마스킹, live 전환 오남용, 외부 호출 차단

1. 시크릿 스캐너는 존재하지만 검사 범위가 좁음

파일:

* trader1/security/source_bundle.py
* trader1/validation/mvp0_validators.py

현재는 source bundle candidate에 포함된 파일만 credential pattern scan을 수행합니다.

문제:

* 실제 zip 전체를 검사하지 않음
* excluded 파일에 시크릿이 있어도 통과 가능
* system/evidence, generated, runtime artifact, pyc 내부는 검사 대상에서 사실상 제외
* binary/pyc 내 문자열 검사 없음

결론:
“후보 소스 번들에는 시크릿이 없다”만 확인합니다. “사용자에게 전달된 패키지 전체에 시크릿이 없다”는 보장하지 못합니다.

2. forbidden_count가 실제 금지 파일 수를 반영하지 않음

파일:

* trader1/security/source_bundle.py

문제:
build_source_bundle_manifest가 항상 다음을 반환합니다.

```text
forbidden_count: 0
```

실제로는 .pyc, **pycache**, generated, system 산출물이 대량 존재합니다.

결론:
보안/위생 validator가 실제 패키지 오염을 실패 처리하지 못합니다.

3. API key 검증은 실제 권한 검증이 아님

파일:

* trader1/security/api_key_permission_check.py

현재 검증은 인자 기반 report builder입니다.

문제:

* 실제 API key를 로드하지 않음
* signed read-only endpoint 호출 없음
* withdrawal 권한 실제 확인 없음
* trade 권한 실제 확인 없음
* Binance key 권한 검증 없음
* IP restriction 검증 없음
* key fingerprint가 실제 key 기반 hash가 아님

현재 값:

```text
api_key_id_hash = "masked-api-key-id"
```

결론:
API key permission check는 보안 스캐폴드이지 실제 권한 검증기가 아닙니다.

4. live preflight는 Upbit 전용

파일:

* trader1/runtime/readiness/live_preflight.py

현재 preflight는 다음으로 고정됩니다.

```text
exchange = UPBIT
market_type = KRW_SPOT
mode = LIVE
```

문제:

* Binance spot live preflight 없음
* Binance futures live preflight 없음
* futures 전용 위험요소 검증 없음
* leverage/margin/position mode 검증 없음

결론:
TRADER_1 전체 범위 기준으로 live review/preflight는 Upbit KRW spot 전용입니다.

5. live config는 강하게 차단하지만 양성 전환 계약이 없음

파일:

* trader1/config/config_schema.py

좋은 점:

```text
live_entry_enabled = false
allow_live_credentials = false
live_order_ready = false
live_order_allowed = false
can_live_trade = false
```

이 값이 true면 즉시 BLOCKED 처리합니다.

문제:
향후 어떤 증거가 충족되면 제한적으로 true가 될 수 있는지에 대한 별도 live promotion config path가 없습니다.

결론:
현재 config는 안전 차단용으로는 좋지만, 실제 live 전환 단계의 양성 경로는 아직 닫혀 있지 않습니다.

6. 로그 마스킹 체계가 별도 모듈로 닫혀 있지 않음

검색 기준으로 전역 redaction/masking logger 계층이 보이지 않습니다.

부족:

* central redactor
* secret-safe logger formatter
* exception sanitizer
* config dump sanitizer
* dashboard artifact sanitizer
* JSON report sanitizer
* test fixtures for redaction

현재 일부 필드에 masked 문자열은 있지만, 시스템 전체 마스킹 정책으로 닫혀 있지는 않습니다.

7. launcher guard의 live marker 검사는 문자열 기반이라 우회 가능

파일:

* trader1/runtime/boot/launcher_guard.py

현재 위험 문자열:

```text
submit_live_order
create_order(
place_order(
send_order(
load_live_credentials
live_api_key
```

문제:

* AST 기반 검사가 아님
* alias/import 우회 가능
* adapter.submit 같은 간접 호출 감지 약함
* ccxt/binance/upbit client 호출 감지 없음
* 문자열 오탐/미탐 가능성 큼

결론:
루트 런처 방어막으로는 유용하지만, live 주문 경로 정적 분석기로는 부족합니다.

8. 외부 네트워크 호출 차단 validator가 부족함

검색상 requests/httpx/urllib/ccxt 사용은 핵심 코드에서 거의 보이지 않습니다. 이 자체는 안전합니다.

문제:
외부 호출이 없음을 강제하는 validator가 명확하지 않습니다.

필요:

* no_network_in_mvp0_mvp4_validator
* no_ccxt_client_validator
* no_exchange_private_call_validator
* allowlisted_read_only_call_validator

결론:
현재는 “없어 보임”이지, “검증기로 금지됨”은 아닙니다.

9. 보안 측면에서 좋은 부분

* 기본 live 플래그는 모두 false
* live preflight는 BLOCKED 고정
* order_adapter_called true를 강하게 차단
* withdrawal_permission_detected는 BLOCKED
* source bundle denylist 개념은 있음
* Authorization Bearer, AKIA, private key block 패턴은 일부 탐지함

10. 우선 패치 목록

11. shipped_package_secret_scan_validator 추가

    * zip 전체 파일 대상
    * included/excluded 구분 없이 검사

12. shipped_package_hygiene_validator 추가

    * .pyc, **pycache**, .env, *.pem, *.key, logs, runtime artifacts 존재 시 FAIL

13. source_bundle.py의 forbidden_count 실제 계산으로 수정

14. secret_scan_validator를 candidate-only와 shipped-package용으로 분리

15. API key permission check를 report builder와 real verifier로 분리

16. Binance spot/futures API key permission schema 추가

17. 전역 redactor 모듈 추가

    * config
    * exception
    * dashboard
    * JSON artifact
    * console output

18. launcher guard를 AST 기반 정적 분석으로 보강

19. no_network/private_api_call validator 추가

20. live promotion path는 별도 evidence-gated config로 분리

특히 source bundle candidate만 깨끗하게 보고 실제 zip 전체 오염을 실패 처리하지 않는 점은 반드시 고쳐야 합니다.
