12차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
코드 품질, 유지보수성, import side effect, 실행 종료성, 도구/validator 구조

1. safe launcher에 무한 콘솔 루프 경로가 있음

파일:
`trader1/runtime/boot/safe_launcher.py`

확인된 함수:
`emit_console_heartbeats`

`ticks is None`이면 Ctrl+C 전까지 계속 반복합니다.

문제:
테스트나 런처 호출 경로에서 ticks 기본값이 None으로 전달되면 종료되지 않습니다. 이전 차수에서 런처/pytest가 제한 시간 내 종료되지 않은 원인 후보로 매우 강합니다.

수정 필요:

* 테스트/검증 모드에서는 ticks=1 강제
* 기본 launcher smoke는 one-shot report만 생성
* continuous monitor는 명시 옵션일 때만 실행

2. tools 디렉터리에 subprocess 기반 emitter가 다수 있음

확인:

* `tools/emit_optimizer_convergence_guardrail_patch_evidence.py`
* `tools/emit_convergence_foundation_patch_evidence.py`
* `tools/emit_root_launcher_operator_visibility_patch_evidence.py`
* `tools/emit_mvp4_external_blocker_report.py`
* 기타 다수

문제:
evidence emitter가 내부에서 subprocess로 pytest/검증기를 다시 실행합니다. 이 구조는 테스트 중첩, timeout, 환경 의존, 느린 재현성 문제를 만듭니다.

권장:

* evidence 생성과 test 실행 분리
* emitter는 이미 생성된 검증 결과를 읽기만 하도록 변경
* subprocess 기반 self-validation은 release 전용 slow 단계로 이동

3. 프로젝트 품질 도구 설정이 없음

루트에서 확인되지 않음:

* `pyproject.toml`
* `pytest.ini`
* `mypy.ini`
* `ruff.toml`
* `requirements.txt`
* `tox.ini`
* `Makefile`

문제:
코드 스타일, 타입검사, 테스트 마커, 의존성, 최소 실행 명령이 닫혀 있지 않습니다.

결과:
다음 AI 구현자가 같은 기준으로 검증하기 어렵습니다.

4. 타입 안정성은 부분적임

일부 함수는 타입 힌트가 있지만, 전체 패키지 기준으로 타입검사 설정이 없습니다.

문제:

* dict[str, Any] 중심 report builder가 많음
* schema와 Python TypedDict/dataclass 연결 약함
* 잘못된 인자 전달이 타입검사에서 잡히지 않음

이미 확인된 실제 사례:
`paper_reconciliation.py`가 존재하지 않는 인자를 `build_reconciliation_report`에 전달합니다.

mypy/pyright가 있었다면 조기에 잡혔을 가능성이 큽니다.

5. 대형 단일 파일 구조가 유지보수성을 크게 해침

대표:

* `trader1/validation/mvp0_validators.py` 약 453KB
* `trader1/dashboard/read_only_dashboard.py` 약 355KB
* `tests/dashboard/test_read_only_dashboard.py` 약 117KB

문제:

* 작은 패치도 광범위 diff 발생
* import 비용 증가
* 특정 validator만 독립 테스트하기 어려움
* AI가 수정 중 주변 로직을 손상시킬 위험 증가

6. placeholder가 runtime/readiness 표면에 남아 있음

확인:
`trader1/runtime/readiness/live_ready_snapshot.py`

예:

* `mvp0-evidence-placeholder`

문제:
blocked snapshot이라도 readiness/live_ready 계층에 placeholder가 남는 것은 위험합니다. live 관련 표면에는 placeholder가 있어도 명시적으로 “BLOCKED_NOT_APPLICABLE” 계열이어야 합니다.

7. validator와 emitter가 누적형 패치 기록처럼 남아 있음

`tools/emit_*_patch_evidence.py`가 대량 존재합니다.

문제:
현재 패키지는 “정리된 소스 패키지”라기보다 “패치 이력과 산출물 생성기가 누적된 작업 디렉터리”에 가깝습니다.

권장:

* tools는 공통 emitter 중심으로 정리
* 과거 patch-specific emitter는 archive 또는 제거
* release source에는 현재 필요한 도구만 포함

8. 정상 상태와 SAFE_MODE 상태가 계속 혼동됨

여러 health/safety 계층에서 blocker가 없어도 final_action이 SAFE_MODE로 나오는 패턴이 있습니다.

문제:
운영자 UI에서 정상 감시 상태와 비상 안전모드가 구분되지 않습니다.

권장:

* 정상: MONITOR_ONLY
* 거래 없음: NO_TRADE
* 차단/위험: SAFE_MODE
* 긴급: KILL_SWITCH 또는 EMERGENCY_REVIEW
