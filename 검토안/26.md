24차 잔여 검토 결과입니다.

판정: 불합격 유지

검토 범위:
예외 처리, 파일 손상 복구, 인코딩, atomic write, 에러 처리 품질

1. JSON 로드가 fail-closed runtime report로 변환되지 않음

`safe_launcher.py`의 `load_json()`은 단순히 다음만 수행합니다.

`json.loads(path.read_text(encoding="utf-8"))`

문제:

* 파일 없음
* JSON 손상
* 인코딩 오류
* 권한 오류

이 상황이 안전한 BLOCKED report로 변환되지 않고 예외로 터질 수 있습니다.

2. write_json / write_text가 fsync 없이 replace만 수행

현재:

* 임시 파일 작성
* replace

부족:

* file flush
* os.fsync
* directory fsync
* 실패 artifact quarantine

전원 장애나 OS crash 기준으로는 durable write가 아닙니다.

3. corrupted JSON artifact 복구 경로가 없음

runtime artifact가 손상되면:

* quarantine
* last-known-good fallback
* repair attempt
* operator guidance

이런 체계가 없습니다.

즉 dashboard/runtime 산출물 하나가 깨지면 부트 경로가 예외로 끊길 수 있습니다.

4. UnicodeDecodeError 처리 범위가 제한적임

`source_bundle.py`는 UnicodeDecodeError를 일부 처리합니다.

하지만 전체 패키지의 JSON/MD/runtime artifact loader는 대부분 UTF-8 디코딩 실패를 공통 정책으로 처리하지 않습니다.

Windows 환경에서 깨진 파일명/문자열/부분 write가 생기면 복구력이 낮습니다.

5. OSError / PermissionError 처리도 제한적임

lock owner PID 읽기 쪽에는 일부 예외 처리가 있습니다.

하지만 중요한 artifact read/write 경로는 권한 오류, 파일 잠금, antivirus lock, cloud sync lock 등을 operator-readable blocker로 변환하지 않습니다.

6. 에러 메시지가 사용자 조치로 연결되지 않음

예:
`runtime artifact writer lock is busy`

이 메시지는 개발자에게는 유용하지만 사용자에게는 다음 정보가 부족합니다.

* 어떤 실행기가 점유 중인지
* stale lock인지
* 기다릴지 종료할지
* 어떤 파일을 확인해야 하는지
* live 주문에는 영향이 없는지

7. schema validator가 파일 손상 상황을 전수 검증하지 않음

schema instance validator는 정상 JSON 객체가 들어온 뒤를 검증합니다.

하지만 다음 단계는 약합니다.

* 파일 열기 실패
* JSONDecodeError
* schema_id 없음
* schema_id unknown
* unsupported schema keyword

이전 차수 결론과 연결됩니다.

8. safe_launcher가 부트 중 예외를 최종 안전 dashboard로 남기지 못함

현재 부트 중 예외가 나면 dashboard/summary/heartbeat 생성이 중단될 수 있습니다.

필요:

* boot_exception_report.json
* final_action = SAFE_MODE
* live flags false
* operator_action = inspect logs
* exception redacted

9. 우선 패치

10. 공통 safe_json_load 추가

11. JSONDecodeError / UnicodeDecodeError / OSError를 BLOCKED report로 변환

12. durable_atomic_write 추가

13. fsync 및 directory fsync 추가

14. corrupted artifact quarantine 추가

15. last-known-good pointer 추가

16. boot_exception_report 생성

17. 에러 메시지에 operator action 포함

18. Windows file lock / PermissionError retry 정책 추가

19. 모든 artifact loader에 공통 예외 처리 적용

24차 점수

예외 처리: 3 / 10
손상 파일 복구: 1.5 / 10
durable write: 3 / 10
Windows 파일 오류 대응: 2 / 10
operator-readable error: 3 / 10

종합: 2.5 / 10

결론

잔여 영역에서도 중요한 결함이 있습니다. 특히 “파일이 깨졌을 때 안전하게 BLOCKED 상태로 dashboard를 남기는 구조”가 약합니다.

실제 장기 운영으로 가려면 market engine 이전에도 공통 파일 IO, 예외 처리, durable write, corrupted artifact quarantine은 먼저 보강해야 합니다.
