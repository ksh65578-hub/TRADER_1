11차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
portfolio, PnL, order intent, WAL, idempotency, execution state machine

1. paper portfolio는 초기 스냅샷 생성기 수준

파일:
`trader1/runtime/portfolio/paper_portfolio.py`

현재 가능한 것:

* 초기 cash/equity snapshot 생성
* UPBIT/KRW_SPOT, BINANCE/SPOT 기본 paper cash 구분
* hash 검증
* live 권한 false 검증

부족:

* ledger event를 받아 cash/position 갱신하는 reducer 없음
* fill event 반영 없음
* fee 차감 없음
* realized/unrealized PnL 계산 없음
* 평균단가 계산 없음
* open/close position lifecycle 없음
* Binance futures portfolio 없음

결론:
“paper portfolio”가 아니라 “paper 초기 잔고 스냅샷”에 가깝습니다.

2. return_pct 계산식이 이상함

검증 함수에서:

`expected_return = (equity - starting + realized) / starting * 100`

그런데 equity가 이미 unrealized와 cash를 포함하고, realized_pnl도 별도 포함되어 있습니다. 일반적인 계좌 equity 기준 수익률이면 realized를 다시 더하면 이중 반영 위험이 있습니다.

현재 초기값에서는 모두 0이라 드러나지 않지만, 실제 realized_pnl이 생기면 계산식이 왜곡될 가능성이 큽니다.

3. paper broker는 주문 체결이 아니라 주문 의도 생성

파일:
`trader1/adapters/upbit/paper_broker.py`

현재 ENTER_LONG이면 생성되는 것은:

* `ORDER_INTENT_CREATED`
* `BUDGET_RESERVED`

없음:

* ORDER_SUBMITTED
* ORDER_ACK_RECEIVED
* ORDER_OPENED
* PARTIAL_FILL_RECEIVED
* ORDER_FILLED
* balance_delta
* position_delta
* fee_amount
* fill price
* order state transition

결론:
paper broker가 실제 paper 체결을 만들지 않습니다. 수익률 검증에 필요한 거래 결과가 생성되지 않습니다.

4. ledger event model은 있으나 state machine 검증이 없음

파일:
`trader1/runtime/ledger/execution_ledger.py`

좋은 점:

* event_hash
* previous_hash
* dedup_key
* duplicate event 차단
* spot short 차단
* scope 분리

부족:

* 주문 상태 전이 순서 검증 없음
* `ORDER_FILLED`가 `ORDER_SUBMITTED` 없이 와도 구조상 통과 가능
* `ORDER_CANCELED` 이후 `ORDER_FILLED` 같은 불가능 상태 차단 없음
* fill 수량 누적 검증 없음
* filled_qty <= order_qty 검증 없음
* reservation과 fill/release의 금액 일치 검증 없음

5. idempotency가 너무 단순함

현재 중복 검사는:

* event_id 중복
* dedup_key 중복
* event_type + intent_id + client_order_id + order_id 중복

문제:
동일 주문의 합법적인 여러 partial fill을 semantic duplicate으로 오탐할 가능성이 있습니다. 반대로 exchange fill id, trade id, execution id가 없어서 실제 중복 체결 이벤트를 정확히 식별하기 어렵습니다.

필요:

* exchange_order_id
* client_order_id
* exchange_trade_id
* fill_id
* order_version
* cumulative_filled_qty
* dedupe key by channel/source

6. intent WAL은 PAPER/UPBIT 전용이며 실제 파일 WAL이 아님

파일:
`trader1/core/events/intent_wal.py`

좋은 점:

* hash chain
* source ledger hash 참조
* scope 검증
* live 권한 false 검증

부족:

* 실제 append-only WAL writer 없음
* fsync 없음
* partial line recovery 없음
* crash 후 replay executor 없음
* Binance scope 없음
* LIVE_REVIEW/LIVE path 없음

결론:
WAL 이벤트 객체 검증기는 있으나, 운영 WAL 계층은 없습니다.

7. position sizing은 안전하지만 실제 주문 수량으로 이어지지 않음

파일:
`trader1/core/sizing/position_sizing.py`

현재 산출:

* selected_notional

부족:

* price 기반 quantity 변환
* tick/step size 반영
* minimum notional 검증
* 주문 가능 잔고 검증
* futures contract quantity 변환
* reduce-only sizing
* liquidation distance 반영

결론:
sizing decision이 실제 executable order로 변환되지 않습니다.

8. Binance portfolio scope가 불완전함

`PAPER_STARTING_CASH_BY_SCOPE`에는 `("BINANCE", "SPOT")`만 있습니다.

없음:

* BINANCE / FUTURES_USDT_M
* futures wallet balance
* margin balance
* unrealized PnL
* maintenance margin
* liquidation price
* funding fee
* long/short position exposure

결론:
Binance futures 1x long/short paper portfolio는 전혀 닫혀 있지 않습니다.

우선 패치 목록

1. `paper_portfolio.py`에 ledger reducer 추가
2. realized/unrealized PnL 계산식 재정의
3. `paper_broker.py`에 simulated fill 생성 추가
4. ledger state machine validator 추가
5. partial fill 누적 검증 추가
6. exchange trade id / fill id 기반 dedupe 추가
7. 실제 JSONL ledger/WAL writer 추가
8. crash recovery용 partial line truncate 추가
9. selected_notional을 quantity/order_request로 변환하는 계층 추가
10. Binance futures portfolio model 추가

11차 점수

* portfolio 구현도: 2 / 10
* PnL 계산 신뢰도: 2 / 10
* paper order simulation: 2 / 10
* ledger hash/idempotency 기초: 6 / 10
* ledger state machine: 2 / 10
* WAL 운영 구현: 2 / 10
* futures portfolio: 0.5 / 10

종합: 3 / 10

결론

현재 portfolio/ledger/order 계층은 “감사 가능한 의도 이벤트 스캐폴드”입니다. 실제 paper trading 성과 검증에 필요한 체결, 잔고, 포지션, PnL, 주문 상태전이, WAL 내구성은 아직 구현되지 않았습니다.
