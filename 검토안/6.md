6차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
거래소 연동 표면, 주문 제약, fee/tick/symbol rule, API key 경계, Binance/Upbit namespace 분리

핵심 추가 발견

1. Upbit symbol rule이 과도하게 단순함

현재 검증은 사실상 이것뿐입니다.

```text
market_type == KRW_SPOT
symbol startswith "KRW-"
symbol split length == 2
```

부족한 항목:

* 업비트 KRW 마켓 실제 종목 목록 검증
* 최소 주문금액 5,000 KRW 검증
* 호가 단위 tick size ladder 검증
* 주문 가능/중지/유의 종목 상태 검증
* 시장가 주문 제약
* quote/base precision
* 주문 수량 반올림/절삭 정책

결론:
symbol rule은 “형식 검증” 수준입니다. 실거래 준비용 제약 검증은 아닙니다.

2. Upbit fee/slippage model이 상수 placeholder 수준

현재:

```text
fee_rate = 0.0005
slippage_bps = 5
```

문제:

* 주문 종류별 수수료 차이 반영 없음
* 예약/TWAP 등 특수 주문 수수료 반영 없음
* maker/taker 구분 없음
* 실시간 spread/orderbook 기반 slippage 없음
* 거래대금/호가잔량 기반 price impact 없음

결론:
paper 손익 계산용 비용 모델로는 부족합니다.

3. paper broker가 거래소 제약을 주문 단계에서 강제하지 않음

현재 paper order 생성은 intent ledger chain 중심입니다.

부족:

* 주문금액 계산
* 최소 주문금액 차단
* tick size 정규화
* 수량 precision 정규화
* fee 차감
* 체결가 산정
* 잔고 변화
* 포지션 변화
* 부분체결
* 주문 거부 사유 taxonomy

결론:
“paper broker”라는 이름과 달리 실제 broker simulation은 아직 구현 전입니다.

4. API key permission check가 Upbit READ_ONLY에만 고정됨

파일:

```text
trader1/security/api_key_permission_check.py
```

현재 scope:

```text
exchange = UPBIT
market_type = KRW_SPOT
mode = READ_ONLY
```

부족:

* Binance spot key 권한 확인 없음
* Binance futures key 권한 확인 없음
* trade permission과 live permission 분리 없음
* IP restriction 검증 없음
* withdrawal 권한 외 권한 세분화 부족
* key fingerprint 산출 방식이 실제 key 기반이 아님
* signed endpoint를 통한 read-only 권한 검증 없음

결론:
API key 안전 스캐폴드는 있으나, 거래소별 실검증 체계는 아닙니다.

5. Live order gateway는 안전하지만 거래소 주문 계약은 없음

좋은 점:

* live order adapter를 절대 호출하지 않음
* final decision 외 직접 주문 차단
* idempotency ambiguous 상태 차단
* ledger/reconciliation/budget reservation 없으면 차단

부족:

* exchange별 order request schema 없음
* Upbit 주문 파라미터 검증 없음
* Binance spot 주문 파라미터 검증 없음
* Binance futures reduceOnly / positionSide / leverage / marginType 검증 없음
* client_order_id exchange별 길이/문자 제한 없음
* retry 정책 없음
* exchange reject code mapping 없음

결론:
현재 live gateway는 “차단기”입니다. 주문 제출 전 검증기 또는 adapter 계약은 아직 아닙니다.

6. Binance root launcher가 futures를 표현하지 못함

현재:

```text
BINANCE_PAPER.py -> MARKET_TYPE = SPOT
BINANCE_LIVE.py -> MARKET_TYPE = SPOT
```

문제:
Binance futures USDⓈ-M 1x long/short 실행 경로가 루트 표면에 없습니다.

권장:

* BINANCE_SPOT_PAPER.py
* BINANCE_SPOT_LIVE.py
* BINANCE_FUTURES_PAPER.py
* BINANCE_FUTURES_LIVE.py

또는 기존 BINANCE_*에서 명확한 market_type 선택 파라미터를 받아야 합니다.

7. namespace 분리는 문서/리포트 중심이고 실제 데이터 경로는 약함

확인된 분리 표면:

* exchange
* market_type
* mode
* session_id
* paper_live_namespace_separated

문제:

* 실제 저장 경로가 exchange/market_type/mode별로 강제되는지 약함
* ledger, snapshot, dashboard, artifact path의 교차오염 방지 validator가 부족함
* Binance가 구현되지 않아 exchange 분리의 실효 검증이 불가능함

8. private stream은 scaffold 수준일 가능성이 큼

파일은 존재합니다.

```text
trader1/adapters/upbit/private_stream.py
```

하지만 전체 구조상 실제 websocket 유지, listen/reconnect, auth renewal, account delta 반영보다는 health/report scaffold 성격입니다.

추가 검토 필요 항목:

* reconnect backoff
* stale stream detection
* sequence gap handling
* account snapshot fallback
* private stream과 reconciliation 연결

현재 패키지 수준에서는 “private stream 운영 구현 완료”로 볼 수 없습니다.

9. 거래소 공식 API 현실 반영 체계가 닫혀 있지 않음

official_api_verification 관련 파일은 있습니다.

하지만 현재 구현은 실제 공식 문서 hash/버전/검증결과 기반의 강제 체계라기보다 report scaffold에 가깝습니다.

부족:

* Upbit 공식 주문 제약 최신성 검증
* Binance exchangeInfo 기반 filter 검증
* Binance futures leverage/margin/position mode 검증
* rate limit/429/418 처리 정책
* server time drift 검증
* signed request canonicalization 검증

6차 점수

* Upbit 거래소 제약 반영: 3 / 10
* Binance 거래소 제약 반영: 1 / 10
* API key 안전 경계: 4 / 10
* Live order 차단 안전성: 8.5 / 10
* 실제 주문 전 검증 완성도: 2.5 / 10
* namespace 분리 실효성: 5 / 10

종합: 4 / 10

결론

현재 거래소 계층은 “실거래 방지 안전 차단”은 강하지만, “실제 거래소 규칙을 반영한 paper/live 준비 계층”은 아직 매우 부족합니다.

특히 Upbit도 아직 형식 검증 중심이고, Binance는 사실상 미구현입니다. 다음 검토는 테스트/CI/패키징/배포 산출물 재현성 쪽으로 이어가는 것이 적절합니다.
