21차 분석 결과입니다.

판정: 불합격 유지

이번 범위:
source package와 runtime/evidence/release 경계

1. system 산출물이 소스 패키지에 과다 포함됨

확인 수량:

* system 전체 파일: 730개
* system/evidence 파일: 673개
* system/runtime 파일: 57개
* system 디렉터리 크기: 약 12MB

문제:
소스 패키지 안에 검증 산출물, audit report, runtime output이 섞여 있습니다.

결론:
현재 압축본은 “소스 패키지”가 아니라 “작업 디렉터리 전체 덤프”에 가깝습니다.

2. runtime 산출물이 실제 실행 증거처럼 포함됨

예:

* system/runtime/upbit/krw_spot/paper/dashboard_shell.json
* system/runtime/upbit/krw_spot/paper/heartbeat.json
* system/runtime/upbit/krw_spot/paper/summary.json
* system/runtime/binance/spot/paper/launcher_report.json
* system/runtime/binance/spot/live/launcher_report.json

문제:
이 파일들이 현재 실행 결과인지, 과거 실행 결과인지, fixture성 산출물인지 구분하기 어렵습니다.

특히 Binance runtime report가 존재하지만 실제 Binance adapter는 없습니다. 사용자가 보면 Binance가 어느 정도 작동하는 것으로 오해할 수 있습니다.

3. futures runtime path가 없음

system/runtime에는 다음만 있습니다.

* binance/spot/live
* binance/spot/paper
* upbit/krw_spot/live
* upbit/krw_spot/paper

없음:

* binance/futures_usdt_m/paper
* binance/futures_usdt_m/live

결론:
runtime 산출물 경계에서도 Binance futures는 전혀 닫혀 있지 않습니다.

4. evidence 파일이 과거 패치 이력처럼 누적됨

system/evidence/audit_reports 안에 MVP4 관련 audit/report가 대량 존재합니다.

문제:

* 어떤 파일이 최신 증거인지 불명확
* 같은 주제의 recheck/audit/final_audit가 중복 존재
* stale evidence를 최신 상태로 오인할 수 있음
* source release에 포함될 필요가 낮음

필요:

* evidence_manifest_current.json
* stale evidence archive 분리
* source package에서는 evidence 제외
* release evidence package는 별도 zip으로 분리

5. contracts/generated도 authority와 혼동 가능

contracts/generated는 약 853KB입니다.

포함:

* ACTIVE_WORKING_VIEW.md
* current_implementation_state.json
* requirement matrix/index
* authority section map
* read cache manifest

문제:
generated/read-cache가 source package에 함께 들어가면 active authority와 혼동됩니다.

권장:
contracts/generated는 “NOT_AUTHORITY_READ_CACHE”로 강하게 표시하거나 release source에서 제외해야 합니다.

6. tools가 실제 runtime보다 큼

크기:

* tools: 약 5.9MB
* trader1: 약 1.6MB

문제:
구현 코드보다 evidence emitter/patch tool이 더 큽니다.

결론:
패키지 중심이 runtime implementation이 아니라 patch/evidence generation으로 치우쳐 있습니다.

7. release package 정의가 필요함

현재는 최소한 네 가지를 분리해야 합니다.

* source_release.zip
* evidence_release.zip
* runtime_artifacts.zip
* generated_read_cache.zip

지금처럼 모두 섞으면 다음 문제가 생깁니다.

* hash identity 불안정
* stale runtime artifact 오인
* pyc/cache 오염
* source review 난이도 증가
* AI 구현자가 과거 증거를 현재 구현으로 오인

8. 우선 패치

9. source package에서 system/runtime 제거

10. source package에서 system/evidence 제거

11. source package에서 pyc/cache 제거

12. Binance futures runtime path absence를 명시적으로 FAIL 처리

13. evidence_current_manifest 추가

14. stale evidence archive 분리

15. contracts/generated에 NOT_AUTHORITY marker 추가

16. release source manifest에 루트 런처와 contracts 포함

17. tools/patch-specific emitter는 release source 제외

18. release package 종류를 명확히 분리

21차 점수

source/runtime 경계: 2 / 10
evidence 최신성: 2.5 / 10
release package 위생: 2 / 10
AI 오인 방지: 3 / 10
runtime artifact 신뢰성: 2.5 / 10

종합: 2.5 / 10

결론

가장 큰 문제는 코드 자체뿐 아니라 패키징 경계입니다. 현재 압축본은 구현 소스, 과거 증거, runtime 산출물, generated read cache, patch 도구가 한꺼번에 섞여 있습니다.

이 상태에서는 추가 구현보다 먼저 source/evidence/runtime/release 패키지를 분리해야 합니다.
