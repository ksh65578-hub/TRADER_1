3차 검토 결과입니다.

판정: 불합격 유지

이번에는 runtime / ledger / reconciliation / crash recovery / paper-live 분리 중심으로 봤습니다.

1. 즉시 수정해야 할 런타임 버그

파일:
trader1/runtime/reconciliation/paper_reconciliation.py

문제:
build_paper_reconciliation_report가 build_reconciliation_report에 존재하지 않는 인자를 넘깁니다.

전달 중인 잘못된 인자:

이건 단순 미구현이 아니라 실제 런타임 파손입니다.

2. reconciliation 계층 자체는 안전하지만 연결부가 깨져 있음

trader1/runtime/reconciliation/reconciliation.py 자체는 비교적 잘 구성되어 있습니다.

좋은 점:

* exchange / market_type / mode / session_id scope 검사
* exchange_snapshot_hash / internal_state_hash 검증
* stale snapshot 차단
* mismatch 발생 시 RECONCILE_REQUIRED
* live permission 필드 재귀 차단
* final_decision을 안전 결정으로 제한

문제:
paper_reconciliation wrapper가 현재 API와 맞지 않아 실제 paper reconciliation 경로가 깨져 있습니다.

즉, core validator는 있는데 호출 계층이 낡은 계약을 사용하고 있습니다.

3. ledger 체인은 MVP 수준에서는 양호하지만 실제 체결 재현에는 부족

파일:
trader1/runtime/ledger/execution_ledger.py

좋은 점:

* event_hash
* previous_hash
* dedup_key
* event_id 중복 차단
* semantic duplicate 차단
* spot short 차단
* entry event에 intent_id 또는 client_order_id 요구

부족:

* 실제 append-only 파일 writer가 보이지 않음
* fsync 기반 내구성 보장 없음
* partial write 복구 로직이 실제 파일 tail truncate 방식으로 닫혀 있지 않음
* exchange order id와 client order id의 lifecycle 매핑이 아직 최소 수준
* partial fill 누적 수량 / 평균단가 / fee 누적 검증이 없음
* Binance futures용 position side, reduceOnly, liquidation, funding 연계가 없음

판정:
MVP 증거 체인으로는 괜찮지만, 실거래 ledger로는 아직 부족합니다.

4. restart recovery는 안전하지만 PAPER 전용에 갇혀 있음

파일:
trader1/core/ledger/restart_recovery.py

좋은 점:

* ledger chain 검증
* intent WAL 검증
* WAL이 ledger event hash를 참조하는지 확인
* live_key_loaded false
* can_submit_order false
* order_adapter_called false
* recovery_action은 RESUME_PAPER_ONLY 또는 SAFE_MODE_RECONCILE

문제:

* UPBIT / KRW_SPOT / PAPER 외 scope는 차단
* 실제 디스크 WAL 읽기/복구가 아니라 report builder 중심
* partial JSONL 손상 복구, lock stale recovery, crash 중복 재실행 방지까지는 실제 runtime로 닫혀 있지 않음

판정:
안전 스캐폴드로는 적합. 실제 장기 운영 복구 계층으로는 미완성.

5. runtime write lock은 있으나 crash durability가 부족

파일:
trader1/runtime/boot/safe_launcher.py

좋은 점:

* .runtime_write.lock 사용
* O_CREAT | O_EXCL 기반 단일 writer lock
* stale lock 처리
* temp file 후 replace 사용

부족:

* 파일 write 후 fsync 없음
* directory fsync 없음
* Windows crash/power loss 상황에서 replace 후 내구성 보장이 약함
* lock 파일 stale 판단이 mtime 기반이라 clock skew나 파일시스템 지연에 취약
* tmp 파일 정리는 있지만 손상된 JSON artifact 복구 전략은 제한적

판정:
동시 write 방지는 일부 되어 있으나, “24/7 장기 운영 crash-safe artifact write”로는 아직 약합니다.

6. heartbeat final_action 논리가 이상함

파일:
trader1/runtime/health/heartbeat.py

현재 build_heartbeat에서:

primary_blocker가 있으면 final_action = NO_TRADE
primary_blocker가 없으면 final_action = SAFE_MODE

정상 heartbeat가 PASS인데 final_action이 SAFE_MODE입니다.

이게 의도된 “항상 안전 모니터링만” 의미라면 허용 가능하지만, 이름상 혼동이 큽니다.

권장:

* PASS heartbeat: final_action = MONITOR_ONLY 또는 NO_TRADE
* BLOCKED heartbeat: final_action = SAFE_MODE 또는 NO_TRADE

현재처럼 정상 상태가 SAFE_MODE로 표시되면 dashboard/operator UX에서 “정상인데 왜 safe mode인가” 혼란이 생깁니다.

파일:

* sizing_decision
* paper dry-run
* restart recovery
* replay consistency
* paper-shadow separation

문제:

* 실제 order fill simulation은 단순 intent chain 수준
* shadow_sample_count가 0으로 고정되는 경로가 있음
* evidence insufficient를 일부 허용하고 PASS_FOR_MVP3_OPERATIONAL_PAPER_ONLY로 지나가는 구조

판정:
운영 루프라기보다 MVP evidence/report orchestration입니다.

결론

현재 패키지는 안전 차단과 보고서 검증 표면은 꽤 넓지만, 실제 runtime engine이라기보다는 “MVP 안전 스캐폴드 + evidence/report generator”에 가깝습니다.

특히 이번에 확인된 paper_reconciliation.py의 TypeError는 즉시 고쳐야 하는 명백한 파손입니다.

다음 검토는 contract / registry / schema / validator 정합성 쪽으로 이어가면 됩니다.
