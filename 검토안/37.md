# 1. 동시성 / 레이스 컨디션 (실전 핵심)

문제:

```text
신호 → 주문 → 체결 → 상태 업데이트
```

이게 항상 순서대로 오지 않는다.

실제 상황:

* 주문 보내기 전에 가격 변함
* 체결 이벤트가 늦게 옴
* 동일 주문 중복 처리
* 포지션이 이미 있는데 또 진입

---

## 반드시 넣어야 할 규칙

```text
single source of truth: position_state

order submit 전에 항상 현재 포지션 재확인
pending order 존재 시 중복 진입 금지
order_id 기준 idempotent 처리
execution event 늦게 와도 상태 깨지지 않도록 설계
```

---

## 차단 조건

```text
포지션 상태 불확실 → 신규 진입 금지
order status 불명확 → 추가 주문 금지
```

---

## Acceptance

```text
어떤 타이밍에서도 중복 진입 / 잘못된 포지션 상태 발생하지 않는다
```

---

# 2. “잘못된 데이터 기반 잘된 판단” 방지

이건 매우 중요하다.

문제:

```text
데이터는 정상처럼 보이는데 실제로는 틀림
```

예:

* 가격 gap
* 거래량 이상치
* 일부 캔들 누락
* WS 끊겼다가 이어짐

---

## 필수 검증

```text
price jump validation
volume spike validation
candle continuity check
event gap detection
```

---

## 차단

```text
이상 데이터 감지 → 해당 구간 no-trade
```

---

# 3. 전략 간 충돌 관리

문제:

```text
pullback: long
breakout: long
VWAP: short (mean reversion)
```

→ 서로 싸움

---

## 규칙

```text
동일 symbol에 대해
동일 방향만 허용 또는
상위 전략 우선순위 적용

conflict 발생 시 no-trade 또는 priority rule 적용
```

---

# 4. 포지션 중첩 정책

문제:

```text
이미 포지션 있는데 추가 진입?
```

---

## 반드시 명시

```text
pyramiding 허용 여부
허용 시 조건 (profit 상태, signal grade 등)
max add count
평균단가 관리 방식
```

---

# 5. 기대값 붕괴 감지 (진짜 중요)

문제:

```text
과거엔 잘됐는데 지금 안됨
```

---

## 감지 조건

```text
rolling expectancy < threshold
profit factor 급락
win rate 급변
slippage 증가
```

---

## 행동

```text
자동 risk-down
전략 비활성화
shadow로 되돌림
```

---

# 6. “거래 안 하는 용기” 강제

이건 문서에 꼭 넣어야 한다.

```text
No-trade is a valid and often optimal decision.
The system must prefer no-trade over low-quality trade.
```

# 7. 목표 왜곡 방지 (최종 핵심)

월 10% 목표가 가장 위험한 요소다.

반드시 넣어라:

```text
If monthly target is not met:
- do NOT increase risk
- do NOT increase frequency
- do NOT loosen filters

Only improvement allowed:
- better selectivity
- better execution
- better capital allocation
```
