5차 검토 결과입니다.

판정: 불합격 유지

이번 범위:
전략, feature, regime, optimizer, replay, paper-shadow 자동개선 루프

1. 전략 엔진이 실제 전략이 아니라 fixture 수준

확인된 전략 파일:

* trader1/core/strategy/strategy_unit.py
* trader1/core/strategy/strategy_level.py

현재 strategy_id 기본값:

* basic_momentum_fixture

문제:
실제 전략 로직이 없습니다.

부족한 항목:

* EMA 50/200 계산
* ADX 계산
* VWAP 계산
* breakout 판정
* pullback 판정
* range mean reversion 판정
* downtrend hard avoidance
* symbol별 regime fit
* candidate score 산출
* 시장 데이터 기반 signal generation

현재는 signal_intent를 외부 인자로 넣으면 ENTER_LONG 또는 NO_TRADE를 담는 구조입니다. 즉 “전략 판단”이 아니라 “전략 판단 결과 모양의 report builder”입니다.

2. feature engineering 구현이 사실상 없음

소스 검색 기준으로 feature 계산 모듈이 없습니다.

없는 핵심 모듈:

* feature builder
* bar aggregator
* EMA/ADX/VWAP calculator
* realized volatility
* vol-of-vol
* CVD proxy
* OFI top-levels
* funding/open interest feature
* regime feature store

결론:
설계도에서 말한 전략/시장상태 판단 재료가 아직 구현되지 않았습니다.

3. regime adaptation은 schema와 validator 중심

관련 schema는 있습니다.

* market_regime_adaptation_report.schema.json
* strategy_condition_matrix.schema.json
* symbol_regime_fit 계열 context 문서

하지만 trader1 runtime/core에 실제 regime classifier는 보이지 않습니다.

결론:
regime은 “계약 표면”만 있고, 실제 bull/bear/range/quiet/squeeze 분류기는 없습니다.

4. optimizer는 실제 최적화기가 아니라 validator/fixture 중심

확인:

* optimizer 관련 schema 다수 존재
* optimizer validator 다수 존재
* optimizer fixture 다수 존재
* trader1 내부 실제 optimizer 실행 모듈 없음

없는 항목:

* parameter search runner
* objective evaluator
* candidate ranking engine
* walk-forward/OOS evaluation runner
* exploration stage A/B controller
* parameter narrowing executor
* optimizer memory writer runtime
* paper 결과 기반 자동 파라미터 갱신기

결론:
자동개선 루프의 안전 계약은 많이 있지만, 실제 최적화 실행기는 아직 없습니다.

5. paper-shadow 루프가 실제 병렬 shadow가 아님

파일:

* trader1/runtime/paper/operational_cycle.py
* trader1/research/shadow/shadow_runner.py
* trader1/research/shadow/evidence_accumulator.py

문제:
operational_cycle.py에서 shadow_sample_count가 0으로 고정됩니다.

shadow_artifact_hash도 실제 shadow 결과가 아니라 “shadow-observation-missing” 기반 해시입니다.

즉 현재는:

* paper dry-run 생성
* shadow separation report 생성
* shadow evidence 부족을 BLOCK_RANKING으로 유지
* 일부 부족 상태는 MVP3 paper-only라서 허용

결론:
사용자가 기대한 “paper 실행 중 shadow 변형들이 병렬로 돌며 수익성 개선 데이터를 축적”하는 구조는 아직 구현되지 않았습니다.

6. replay도 실제 시장 재현 엔진이 아님

파일:

* trader1/research/replay/replay_runner.py

현재 replay는 input_events의 해시 반복 일치성을 보는 deterministic consistency report에 가깝습니다.

부족:

* historical market event stream 재생
* warm replay
* overlap dedupe check
* bar/orderbook/aggTrades 재구성
* strategy rerun
* fill simulation
* replay 결과 성과 측정
* replay와 optimizer 연결

결론:
replay는 “결정성 검증 스캐폴드”이지, 전략 검증용 replay engine은 아닙니다.

7. position sizing은 안전하지만 매우 단순

좋은 점:

* equity/cash/risk/liquidity/depth cap의 min-of-caps 구조
* live/order permission 생성 차단
* missing input 차단

부족:

* exchange별 tick/min-notional 반영 약함
* futures margin/liquidation 반영 없음
* strategy/regime별 risk scaling 없음
* drawdown regime 연동 없음
* Kelly류 과최적화 방지는 있으나, 실제 기대값 기반 sizing은 없음

판정:
MVP paper safety sizing으로는 가능. 수익성 최적화 sizing은 아님.

8. paper broker는 “체결 시뮬레이터”가 아니라 intent ledger 생성기

파일:

* trader1/adapters/upbit/paper_broker.py

좋은 점:

* live key 미사용
* order adapter 차단
* symbol rule, fee, slippage baseline 확인
* paper ledger events 생성
* entry/no-trade reason 기록

부족:

* 실제 fill price 계산 없음
* bid/ask/orderbook 기반 체결 없음
* partial fill 없음
* balance update 없음
* position update 없음
* realized/unrealized PnL 없음
* fee 차감 없음
* slippage가 단순 baseline 표기 수준
* portfolio snapshot과 실제 연결 약함

결론:
paper broker라기보다 paper intent dry-run입니다.

9. candidate generation 구현 없음

schema는 있습니다.

* candidate_generation_report.schema.json
* candidate_scorecard.schema.json
* strategy_candidate.schema.json

하지만 실제 candidate universe selector는 확인되지 않았습니다.

없는 항목:

* 거래대금/유동성 기반 종목 필터
* 변동성 필터
* stablecoin/저변동 제외
* 동적 universe rotation
* exchange/market_type별 candidate 분리
* candidate scorecard producer

10. 설계 의도와 현재 구현의 차이

설계 의도:

* Paper를 켠다
* Paper/Shadow가 데이터를 쌓는다
* 전략과 파라미터가 자동 개선된다
* 충분한 증거가 쌓이면 LIVE_READY 후보가 된다
* 사용자는 dashboard 확인 후 live 실행기로 전환한다

현재 구현:

* Upbit paper dry-run report 생성
* strategy/sizing/replay/shadow/recovery report 조합
* live는 계속 차단
* shadow evidence는 부족 상태
* optimizer는 validator/fixture 중심
* 실제 시장 데이터 루프와 자동 개선기는 없음

5차 점수

전략 구현도: 2 / 10
feature 구현도: 1 / 10
regime 구현도: 2 / 10
optimizer 실행 구현도: 2 / 10
paper-shadow 자동개선 루프: 2.5 / 10
replay engine 구현도: 2.5 / 10
safety guardrail: 8 / 10

종합: 4 / 10

결론

현재 패키지는 “수익성 개선형 자동매매 시스템”이 아니라 “수익성 개선 시스템을 만들기 위한 계약, schema, validator, 안전 차단 스캐폴드” 단계입니다.
