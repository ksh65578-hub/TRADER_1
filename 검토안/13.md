13차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
문서-코드 추적성, active authority surface, authority manifest, generated/context pack, current implementation state

1. authority manifest가 현재 registry와 불일치

`contracts/authority_manifest.json`에는 registry 생성 당시 hash가 다음으로 기록되어 있습니다.

`7DD30427409029450DAAC6189131869F33991979C2232C3AC86B122ABA20870D`

하지만 현재 `contracts/registry.yaml` 실제 SHA256은 다음입니다.

`681809388927F772D999C9B420C6791D70DA3784F97A80D90172288D8C71581C`

결론:
현재 authority manifest는 registry 최신 상태를 증명하지 못합니다.

이 상태에서는 문서 기준상 최소한:

* `authority_hash_checked=false`
* live readiness claim 차단
* release identity PASS 차단
* live-affecting patch 차단

으로 봐야 합니다.

2. TRADER_1.md와 AGENTS.md hash는 manifest와 일치

확인:

* TRADER_1.md hash 일치
* AGENTS.md hash 일치

따라서 문서 본문 자체의 manifest mismatch는 아닙니다.

문제는 registry가 manifest 생성 당시 상태와 달라졌다는 점입니다.

3. generated artifact가 너무 많이 포함되어 active authority와 혼동됨

`contracts/generated/context_pack/*`가 대량 포함되어 있습니다.

문서상 generated navigation/context pack은 authority가 아니라 read cache입니다.

하지만 패키지 구조상:

* active working view
* context pack
* authority section map
* current implementation state
* audit reports
* runtime artifacts

가 모두 소스 패키지에 섞여 있습니다.

문제:
AI 구현자가 generated cache를 실제 authority로 오인할 가능성이 큽니다.

4. `ACTIVE_WORKING_VIEW.md`가 현재 구현 상태를 과도하게 요약함

`contracts/generated/ACTIVE_WORKING_VIEW.md`는 `current_mvp: MVP-4`로 표시합니다.

하지만 실제 코드 상태는 앞선 검토 기준:

* market data collector 없음
* strategy/feature 구현 없음
* Binance adapter 없음
* paper fill simulator 없음
* live ready snapshot writer 양성 경로 없음
* registry manifest mismatch

입니다.

결론:
working view가 “진행 관리 요약”으로는 가능하지만, 구현 완료성 판단 근거로 사용되면 위험합니다.

5. `current_implementation_state.json`도 과대 표시 위험이 큼

파일은 `current_mvp: MVP-4`와 다수의 completed requirement를 포함합니다.

문제:
completed requirement가 실제 runtime 구현 완료인지, schema/validator/scaffold 완료인지 구분하지 않습니다.

예:

* `REQ-MVP1-EXECUTION-LEDGER-SCAFFOLD`
* `REQ-MVP1-RECONCILIATION-SCAFFOLD`
* `REQ-MVP3-OPERATIONAL-UPBIT-PAPER-FOUNDATION`

이런 항목은 이름상 scaffold/foundation인데, 패키지 전체 상태에서는 MVP-4 진행처럼 보입니다.

필요:

* scaffold_complete
* validator_complete
* runtime_producer_complete
* runtime_consumer_complete
* integration_tested
* exchange_implemented
* live_blocked_only

로 분리해야 합니다.

6. active authority와 retained archive 경계는 문서에 있으나 코드 검증이 약함

TRADER_1.md와 AGENTS.md에는 다음 원칙이 반복됩니다.

* active contract pack 우선
* retained archive는 실행 authority 아님
* generated navigation artifact는 authority 아님
* conflict 시 fail-closed 우선

문제:
코드/validator가 실제로 다음을 강제하는지는 약합니다.

* generated context pack을 authority로 쓰지 않았는지
* retained archive-only 요구가 live permission으로 승격되지 않았는지
* current_implementation_state가 authority처럼 사용되지 않았는지
* stale authority_section_map 사용을 차단하는지

7. source tree hash도 manifest와 불일치 가능성이 큼

authority manifest에는 `source_tree_hash_when_generated`가 있습니다.

하지만 현재 패키지는:

* pyc 포함
* generated/runtime/evidence 포함
* registry hash mismatch
* 산출물 대량 포함

상태입니다.

따라서 source tree hash는 현재 zip의 실제 상태를 대표하지 못할 가능성이 큽니다.

8. 문서 자체는 “안전한 해석 규칙”을 많이 포함하지만, 코드가 그만큼 못 따라옴

문서에는 좋은 규칙이 많습니다.

* generated cache is not authority
* authority hash mismatch blocks live
* retained archive cannot weaken live safety
* active contract pack first
* fail-closed priority

하지만 실제 코드 상태는:

* manifest mismatch를 즉시 package FAIL로 승격하는 경로가 약함
* generated artifacts가 release/source에 섞임
* current implementation state가 과대 표시됨
* scaffold와 runtime complete 구분이 약함

입니다.

9. 이번 차수 우선 패치 목록

10. `authority_manifest_validator`가 registry 현재 hash mismatch를 FAIL로 처리

11. registry 수정 후 authority manifest 재생성

12. source tree hash 재계산 및 manifest 갱신

13. generated cache와 source authority를 패키징 단계에서 분리

14. `current_implementation_state.json`에 completion kind 추가

15. `ACTIVE_WORKING_VIEW.md` 상단에 “NON_AUTHORITY_READ_CACHE” 명시

16. context pack 전체에 “NOT_AUTHORITY” marker 강제

17. stale authority_section_map 차단 validator 추가

18. generated/read-cache 파일이 validator 입력 authority로 쓰이지 않았는지 검사

19. release identity PASS 조건에 registry hash 일치 추가

13차 점수

* 문서 해석 규칙 품질: 8 / 10
* authority manifest 현재성: 3 / 10
* generated/source 경계: 3 / 10
* 구현 상태 추적 정확도: 3.5 / 10
* active authority 강제력: 5 / 10
* AI 구현 오인 방지: 4 / 10

종합: 4.5 / 10

결론

문서 자체의 authority 원칙은 비교적 잘 설계되어 있습니다. 문제는 현재 패키지가 그 원칙을 산출물/코드/검증 단계에서 충분히 강제하지 못한다는 점입니다.

특히 registry hash mismatch는 단순 정리 문제가 아니라 release identity와 live readiness를 막아야 하는 authority 결함입니다.
