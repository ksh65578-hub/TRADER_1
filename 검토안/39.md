있다. 하지만 이제는 “더 넣어야 한다”보다 “너무 많이 넣어서 구현자가 흔들리지 않게 정리해야 하는 항목”이다.

추가 고려는 5개 정도만 남긴다.

# 1. 우선순위 체계

모든 규칙이 동시에 있으면 AI가 헷갈릴 수 있다.

반드시 우선순위를 고정해야 한다.

```text
1. safety
2. data integrity
3. ledger consistency
4. risk limit
5. execution quality
6. strategy signal
7. profitability target
```

수익률 목표는 항상 마지막이다.

# 2. 충돌 시 기본 행동

규칙끼리 충돌하면 기본값을 정해야 한다.

```text
충돌 발생 시 no-trade
상태 불확실 시 no-trade
데이터 불확실 시 no-trade
포지션 불확실 시 신규 진입 금지
live 판단 불확실 시 live 차단
```

# 3. 최소 구현 순서

좋은 항목이 많아도 순서를 틀리면 실패한다.

```text
1. registry/schema/validator
2. exchange namespace
3. paper ledger
4. data health
5. risk engine
6. strategy signal
7. shadow competition
8. LIVE_READY snapshot
9. micro-live
10. scaled live
```

# 4. 문서 비대화 방지

이제 더 추가하면 중복과 모순이 생길 수 있다.

따라서 문서에는 이렇게 넣는 게 좋다.

```text
상세 전략 수식은 strategy contract에 둔다.
실행 금지/허용 규칙은 AGENTS에 둔다.
schema/key/enum은 registry에 둔다.
acceptance는 test manifest에 둔다.
```

# 5. 최종 경고 문장

AI compiler에게 반드시 넣을 문장이다.

```text
Do not invent missing live behavior.
If a rule is missing, implement the safest scaffold, block live use, and record the missing contract.
```

최종 결론:

```text
추가로 더 넣을 수는 있지만,
이제 핵심은 추가가 아니라 정리, 우선순위, 충돌 해결, 구현 순서 고정이다.
```

이 정도면 설계도와 지침서에는 충분하다.
