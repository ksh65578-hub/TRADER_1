20차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
placeholder/stub 잔존, mock-safe 표현, 문서와 evidence 간 불일치 가능성

1. placeholder 관련 문구가 아직 대량 존재

검색 결과 `AGENTS.md`에 placeholder 관련 지시가 다수 남아 있습니다.

예:

* `fixture catalog placeholders`
* `registry_placeholders_remaining`
* `authority_sha256 placeholder`
* `remove registry-defined-required and all other placeholders`
* `placeholder projection`

문서에서 “placeholder 제거 필요”를 지시하는 문맥도 있지만, 구현자 입장에서는 현재 패키지가 placeholder 정리 완료 상태인지 아닌지 혼동될 수 있습니다.

2. evidence는 placeholder 없음이라고 주장함

다수 patch_result 파일은 다음 값을 가집니다.

```json
"registry_placeholders_remaining": []
```

문제:
문서와 registry에는 placeholder 금지/placeholder 관련 표면이 계속 남아 있고, evidence는 “없음”이라고 주장합니다.

이건 실제 placeholder가 남았다는 뜻이라기보다, placeholder 검사 범위가 registry 일부에 한정되어 있을 가능성이 큽니다.

3. mock-safe 표현이 active 문서에 남아 있음

확인된 표현:

* `REPLAY/PAPER/SHADOW/READ_ONLY mock-safe checks only`
* `adapter may be implemented as contract or mock`
* `adapter mock added`

문제:
mock 허용은 초기 scaffold 단계에서는 가능하지만, 현재 패키지가 MVP-4처럼 표시되는 상태에서는 mock과 실제 구현의 경계가 훨씬 강하게 표시되어야 합니다.

4. registry에는 placeholder 금지 플래그가 있음

`contracts/registry.yaml`에 다음 표면이 있습니다.

```json
"placeholder_hash_forbidden": true
```

좋은 점:
placeholder 금지 의도는 존재합니다.

문제:
이 플래그가 실제 source 전체, evidence 전체, generated artifact 전체, dashboard 전체에 적용되는지 약합니다.

5. patch_result가 너무 방대하고 신뢰 경계가 불명확함

patch_result 파일이 대량 존재하며, 각 파일이 매우 큽니다.

문제:

* 어떤 patch_result가 현재 authoritative evidence인지 불명확
* 과거 patch_result가 최신 상태와 충돌할 수 있음
* evidence package와 source package가 섞여 있음
* AI가 오래된 patch_result를 최신 증거로 오해할 수 있음

6. 문서상 “mock 가능”과 사용자 목표 사이의 간극

사용자 목표:

* Paper를 돌려 수익성 검증
* 충분하면 live 전환
* Upbit + Binance spot/futures 통합

현재 mock-safe 문구:

* scaffold/mock adapter 허용
* read-only/mock-safe check 허용

결론:
mock 허용 문구는 MVP-0/MVP-1에는 필요하지만, 이후 단계에서는 반드시 “mock은 live/paper profitability evidence로 인정 금지”로 강제해야 합니다.

7. 우선 패치 목록

8. placeholder scan 범위를 전체 source/evidence/generated로 확장

9. placeholder 허용 문맥과 금지 문맥을 분리

10. `registry_placeholders_remaining` 외에 `source_placeholders_remaining`, `evidence_placeholders_remaining` 추가

11. mock adapter는 `SCAFFOLD_ONLY`로만 허용

12. mock-generated evidence는 profitability/live readiness 증거로 사용 금지

13. current implementation state에 `mock_or_fixture_backed` 필드 추가

14. patch_result 최신성 manifest 추가

15. stale patch_result는 release source에서 제외

16. evidence package와 source package 분리

17. `MVP-4` 표시는 scaffold/runtime/exchange 구현 성숙도와 분리

20차 점수

placeholder 관리: 4 / 10
mock/fixture 경계: 3 / 10
evidence 최신성: 3 / 10
source/evidence 분리: 2.5 / 10
AI 오인 방지: 3.5 / 10

종합: 3.5 / 10

결론

현재 패키지는 placeholder와 mock 자체보다, “어디까지가 허용된 scaffold이고 어디부터 실제 구현 증거인지”가 불명확한 것이 더 큰 문제입니다.

이제 추가 검토 효율은 낮습니다. 다음 단계는 중복 제거된 최종 결함표와 패치 우선순위 작성이 맞습니다.
