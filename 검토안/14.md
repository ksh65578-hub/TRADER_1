14차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
테스트 품질, fixture 의존성, 자기충족형 validator, 결함 은폐 가능성

1. 테스트가 실제 런타임보다 report builder를 검증함

확인된 테스트 예:
`tests/integration/test_upbit_operational_paper_cycle.py`

이 테스트는 실제 paper trading 루프가 아니라 `build_upbit_operational_paper_cycle()`이 만든 report를 다시 `validate_paper_operation_gate_report()`로 검증합니다.

문제:

* 시장 데이터 입력 없음
* 주문 체결 없음
* 포트폴리오 변화 없음
* shadow 실행 없음
* replay 실행 없음
* optimizer 실행 없음

결론:
통합 테스트라는 이름이지만 실제 통합 런타임 테스트가 아니라 report composition test입니다.

2. adapter 테스트도 fixture 중심

확인된 테스트:
`tests/adapter/test_upbit_paper_adapter.py`

검증 대상:

* `build_upbit_public_market_data_fixture`
* `validate_upbit_public_market_data`
* `validate_upbit_krw_symbol`
* `build_upbit_paper_dry_run_report`

문제:
실제 Upbit adapter 동작이 아니라 fixture/report 검증입니다.

부족:

* 실제 market metadata parsing
* orderbook/trade payload parsing
* tick size 검증
* min notional 검증
* reject reason mapping
* fill simulation

3. root launcher 테스트가 Binance futures 누락을 잡지 못함

`tests/contract/test_root_launchers.py`는 정확히 네 개 런처 존재를 요구합니다.

허용 런처:

* UPBIT_PAPER
* UPBIT_LIVE
* BINANCE_PAPER
* BINANCE_LIVE

문제:
이 테스트 구조 자체가 Binance spot/futures 분리를 요구하지 않습니다. 따라서 Binance futures 실행 표면이 없어도 PASS할 수 있습니다.

필요:

* Binance market_type coverage test
* FUTURES_USDT_M launcher or parameter path test
* futures live preflight blocked test
* futures paper scaffold test

4. 테스트가 “PASS_FOR_MVP…” 상태를 성공으로 고정함

예:
`PASS_FOR_MVP3_OPERATIONAL_PAPER_ONLY`

문제:
이 상태는 실제 구현 완료가 아니라 “MVP paper-only scaffold 통과”에 가깝습니다. 그런데 테스트는 이를 정상 성공으로 고정합니다.

결과:
향후 실제 구현으로 확장되어야 할 영역이 “이미 성공”처럼 굳어질 위험이 있습니다.

5. negative test는 live 차단에 집중되어 있음

잘 되어 있는 부분:

* live_order_ready 변조 차단
* live_order_allowed 변조 차단
* can_live_trade 변조 차단
* order adapter 호출 차단
* scope mismatch 차단

부족한 negative test:

* paper fill 없이 PnL 생성 금지
* shadow sample 0인데 ranking 허용 금지
* Binance futures path 누락 금지
* source zip에 pyc 포함 시 실패
* registry hash mismatch 시 release identity 실패
* READY 용어 오남용 금지
* schema instance와 실제 runtime artifact 불일치 실패

6. fixture가 과도하게 많고 실제 producer가 없음

`tests/validators/fixtures`에 optimizer/convergence/regime/profitability 관련 fixture가 대량 존재합니다.

문제:
fixture는 많지만 해당 fixture를 생성하는 실제 runtime producer가 없습니다.

결과:
validator는 “손으로 만든 좋은 JSON/나쁜 JSON”은 판별하지만, 시스템이 그 JSON을 실제로 만들 수 있는지는 증명하지 못합니다.

7. 테스트가 결함을 숨기는 대표 패턴

현재 패턴:

* builder가 JSON 생성
* 같은 모듈 validator가 JSON 검증
* 테스트는 status가 PASS인지 확인

문제:
builder와 validator가 같은 잘못된 가정을 공유하면 테스트가 통과합니다.

필요:

* independent golden schema validation
* real file artifact validation
* producer-consumer integration test
* mutation/fuzz test
* cross-module contract test

8. test naming이 구현 성숙도를 과장함

예:

* `test_upbit_operational_paper_cycle`
* `test_upbit_paper_adapter`
* `test_read_only_dashboard`
* `test_emergency_flatten_dry_run`

이름만 보면 실제 운영/adapter/emergency 기능처럼 보이지만, 실제로는 dry-run/report 검증이 대부분입니다.

권장:

* `test_upbit_paper_report_scaffold`
* `test_upbit_market_data_fixture`
* `test_emergency_flatten_dry_run_report`
* 실제 기능 구현 후 현재 이름 사용

결론

테스트 파일은 많지만, 대부분은 실제 자동매매 런타임 검증이 아니라 스캐폴드 report/fixture 검증입니다.
현재 테스트 통과는 “안전 차단 report가 형식상 만들어진다”는 의미에 가깝고, “paper trading, shadow learning, replay, optimizer, exchange adapter가 작동한다”는 증거가 아닙니다.
