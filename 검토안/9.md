9차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
config, mode state, readiness, LIVE_READY snapshot, live review dashboard, 사용자 오인 가능성

1. config는 fail-closed지만 상태전이 모델이 없음

파일:
trader1/config/config_schema.py

좋은 점:

* live_entry_enabled=false
* allow_live_credentials=false
* live_order_ready=false
* live_order_allowed=false
* can_live_trade=false
* true 시도 시 BLOCKED
* exchange/market_type/mode enum 검증
* Binance market_type은 명시 selector 요구

문제:
현재 config는 “무조건 live 금지” 모델입니다.

부족한 항목:

* PAPER -> SHADOW -> LIVE_REVIEW -> SMALL_LIVE_BURN_IN -> LIVE_ACTIVE 상태전이 모델
* 상태전이별 요구 증거
* live 가능 config와 일반 runtime config의 분리
* 누가 언제 어떤 증거로 live_order_ready를 true로 만들 수 있는지에 대한 양성 경로

결론:
안전 차단은 좋지만, live 전환 가능한 완성 시스템의 config 모델은 아직 아닙니다.

2. LIVE_READY snapshot writer는 구조상 항상 차단됨

파일:
trader1/runtime/readiness/live_ready_snapshot.py

evaluate_live_ready_snapshot_writer는 많은 조건을 검사하지만 마지막에 다음처럼 끝납니다.

* allow_snapshot_write가 false면 차단
* allow_snapshot_write가 true여도 “LIVE_READY snapshot creation is not enabled by this MVP-0 implementation”으로 차단

즉 현재는 어떤 증거를 넣어도 snapshot 생성이 불가능합니다.

판정:
MVP 안전 스캐폴드로는 타당합니다. 하지만 “paper/shadow 성과 누적 후 LIVE_READY 후보 생성” 요구사항 기준으로는 미구현입니다.

3. blocked snapshot에 placeholder가 남아 있음

build_blocked_live_ready_snapshot 기본값:

* snapshot_id = mvp0_blocked_snapshot
* strategy_id = mvp0_strategy
* strategy_build_id = mvp0_build
* parameter_hash = mvp0_parameter_hash
* manifest_hash = mvp0-blocked-manifest

문제:
blocked snapshot이라도 dashboard/evidence 표면에 노출되면 실제 전략/파라미터 증거처럼 보일 수 있습니다.

권장:
placeholder 값은 “NOT_APPLICABLE_BLOCKED_SNAPSHOT”처럼 오해 가능성이 낮은 값으로 바꿔야 합니다.

4. live_preflight는 Upbit live review 전용

파일:
trader1/runtime/readiness/live_preflight.py

검증 조건:

* exchange == UPBIT
* market_type == KRW_SPOT
* mode == LIVE

문제:

* Binance spot live review 없음
* Binance futures live review 없음
* futures 전용 live blocker 없음
* market_type별 preflight 분기 없음

결론:
TRADER_1 전체 범위 기준으로 live review 표면이 Upbit에만 닫혀 있습니다.

5. live_review_dashboard도 Upbit KRW spot 전용

파일:
trader1/dashboard/live_review_dashboard.py

검증 조건:

* exchange == UPBIT
* market_type == KRW_SPOT
* mode == LIVE

문제:
Binance spot/futures dashboard 검증 경로가 없습니다.

결론:
dashboard live review는 “통합 시스템” 표면이 아니라 Upbit live review 표면입니다.

6. readiness_surface의 render_first_line은 PAPER에 과도하게 긍정적일 수 있음

파일:
trader1/runtime/readiness/readiness_surface.py

PAPER 모드에서:
“PAPER TRADING: READY - live orders blocked”

문제:
현재 paper 구현은 실제 시장 기반 paper trading이 아니라 fixture/report dry-run에 가깝습니다.

따라서 “PAPER TRADING: READY”는 사용자에게 실제 paper trading 준비 완료로 오해될 수 있습니다.

권장 표현:

* “PAPER SCAFFOLD: READY - live orders blocked”
* “PAPER DRY-RUN: READY - live orders blocked”
* 실제 market data loop, fill simulator, portfolio update가 붙은 뒤에만 “PAPER TRADING: READY” 허용

7. readiness_surface는 can_* 상태와 실제 producer 구현을 검증하지 않음

build_readiness_surface는 인자로 can_collect_data, can_paper_trade, can_shadow_evaluate 등을 받습니다.

문제:
이 값이 true일 때 실제 producer 존재와 연결되는 검증이 없습니다.

예:

* can_paper_trade=true인데 실제 fill simulator 없음
* can_shadow_evaluate=true인데 shadow_sample_count=0
* can_replay=true인데 replay가 deterministic hash check 수준

결론:
readiness surface는 상태 표시 객체이지, 실제 구현 보증 객체가 아닙니다.

8. dashboard는 오인 방지 문구가 많지만 상태명 일부가 과함

좋은 점:
read_only_dashboard.py에는 “not LIVE_READY”, “live orders blocked”, “scale-up blocked” 문구가 매우 많이 들어 있습니다.

문제:
동시에 다음 상태명이 존재합니다.

* PAPER_REVIEW_READY
* SCORECARD_INPUT_READY
* READY_FOR_PAPER_RANKING_REVIEW
* PROFITABILITY_PROGRESS_STATUS READY

이들은 live_ready와 구분되지만, UI에서 READY 단어가 과도하게 많습니다.

권장:

* READY 대신 REVIEWABLE, INPUT_COMPLETE, RANKING_REVIEWABLE 사용
* “LIVE_READY”와 다른 ready 용어를 최대한 줄이기

파일:
trader1/dashboard/live_review_dashboard.py

FORBIDDEN_WORDING에는 “LIVE READY”가 포함됩니다.

build_live_review_dashboard에서:
forbidden_wording_present = any(text.lower() in first_line.lower() ...)

문제:
검사 대상이 first_line뿐입니다.

dashboard 전체 blockers/message/operator warning에는 forbidden wording이 있어도 탐지하지 못합니다.

필요:

* dashboard 전체 JSON 문자열 대상 검사
* HTML 렌더링 결과 대상 검사
* summary_writer 산출물 대상 검사

10. summary_writer는 live_ready 객체를 만들지만 이름이 다소 위험함

파일:
trader1/dashboard/summary_writer.py

summary 내부에 live_ready 섹션이 존재합니다.

검증상 live_order_ready/live_order_allowed true는 막지만, 명칭 자체가 “live_ready”라서 blocked 상태와 혼동될 수 있습니다.

권장:

* live_ready_status
* live_block_status
* live_readiness_gate

11. 상태 명칭과 실제 구현 성숙도 사이의 불일치

현재 코드에는 “MVP-4”, “paper operational cycle”, “live review dashboard”, “PAPER_REVIEW_READY” 같은 용어가 많습니다.

하지만 실제 구현은:

* fixture/report 중심
* 실제 data loop 부족
* paper fill simulator 부족
* shadow 병렬 평가 부족
* optimizer 미구현
* Binance 미구현

결론:
상태명은 구현보다 앞서 있습니다. AI 구현자와 사용자가 진행도를 과대평가할 위험이 있습니다.

우선 패치 목록

1. readiness state를 구현 성숙도별로 분리

* SCAFFOLD_READY
* DRY_RUN_READY
* PAPER_RUNTIME_READY
* PAPER_REVIEWABLE
* LIVE_REVIEW_BLOCKED
* LIVE_READY_CANDIDATE

2. “READY” 단어 사용 제한

* LIVE_READY 외 ready 용어 축소
* PAPER_REVIEW_READY -> PAPER_REVIEWABLE
* SCORECARD_INPUT_READY -> SCORECARD_INPUT_COMPLETE

3. LIVE_READY snapshot writer에 양성 경로 설계 추가

* 단, live_order_allowed와 분리
* snapshot 생성 가능과 주문 허용을 분리

4. blocked snapshot placeholder 값을 오해 불가능한 값으로 변경

5. Binance spot/futures live_preflight scaffold 추가

6. live_review_dashboard를 exchange/market_type parameterized 구조로 변경

7. forbidden wording 검사를 dashboard 전체 payload와 HTML 결과로 확장

8. can_* readiness flag가 true일 때 producer/validator 존재를 강제

9. summary_writer의 live_ready 섹션명을 live_block_status로 변경

10. UI 첫 줄에서 “PAPER TRADING READY”를 실제 paper runtime 완성 전까지 금지

9차 점수

* live 차단 상태표현: 8 / 10
* 사용자 오인 방지: 6 / 10
* LIVE_READY 양성 경로: 1 / 10
* mode/state 전이 모델: 3 / 10
* dashboard truthfulness: 5.5 / 10
* exchange 범위 포괄성: 2 / 10

종합: 4.5 / 10

결론

현재 readiness/dashboard 계층은 “live는 절대 안 된다”는 메시지는 강합니다. 하지만 “무엇이 실제로 준비됐고, 무엇은 스캐폴드일 뿐인지”를 충분히 구분하지 못합니다.

특히 READY 계열 용어가 과도하고, LIVE_READY snapshot은 어떤 경우에도 생성되지 않는 구조라서 자동 개선 후 live 후보 생성이라는 목표에는 아직 도달하지 못했습니다.
