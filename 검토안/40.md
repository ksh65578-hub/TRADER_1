이제는 “더 추가”의 영역이 아니라 **마무리(closure) 조건을 명시하는 단계**다.
그래도 실전에서 마지막으로 빠지기 쉬운 항목 몇 개만 압축해서 정리한다.

---

# 1. 종료(Shutdown) 시맨틱

문제: 중단 시 상태가 꼬이면 이후 모든 검증이 무의미해진다.

필수 규칙

```text
graceful_shutdown: 신규 진입 즉시 차단 → 미체결 정리 → 스냅샷 저장 → 종료
force_shutdown: 상태 불확실 시 kill switch 유지, 재시작 후 복구 절차 강제
```

Acceptance

```text
중단/재시작을 반복해도 포지션/잔고/ledger 불일치가 없다.
```

---

# 2. 시간/클럭 일관성

문제: 시계 드리프트는 체결·펀딩·캔들 정렬을 깨뜨린다.

필수 규칙

```text
NTP 동기화
exchange server time과 주기적 비교
드리프트 초과 시 no-trade
```

---

# 3. 버전 고정과 재현성 패키징

문제: 코드/파라미터가 바뀌면 성과 비교가 무의미해진다.

필수 규칙

```text
release_id, commit_hash, config_hash 고정
LIVE_READY snapshot에 version 포함
replay는 해당 버전으로만 재현
```

---

# 4. 데이터 보존/보관 정책

문제: 로그 과다/부족 모두 문제.

필수 규칙

```text
raw/canonical/ledger 분리 저장
핵심 이벤트는 장기 보존, 대용량 원본은 롤링 보관
보존기간 정책(예: 30/90/180일) 명시
```

---

# 5. 리소스 한계 관리

문제: 메모리/디스크/CPU로 시스템이 멈춘다.

필수 규칙

```text
queue/backlog 상한
메모리 상한 초과 시 oldest drop 또는 rate limit
디스크 임계치 도달 시 로그 축소/아카이브
```

---

# 6. 멀티프로세스/멀티인스턴스 충돌 방지

문제: 동일 전략이 두 번 실행되면 중복 주문 발생.

필수 규칙

```text
single-writer lock(파일/DB/redis 등)
instance_id 기반 리더 선출
중복 실행 감지 시 보조 인스턴스는 read-only
```

---

# 7. 구성 변경 관리(Hot reload 경계)

문제: 런타임 중 설정 변경이 상태를 깨뜨림.

필수 규칙

```text
핵심 파라미터는 런타임 변경 금지
허용된 항목만 hot reload
변경 시 snapshot 버전 증가 및 audit 기록
```

---

# 8. 샌드박스/시뮬레이션 격리

문제: 테스트 코드가 실거래 경로에 영향을 준다.

필수 규칙

```text
replay/paper/shadow/live 완전 격리된 namespace
테스트용 키/엔드포인트 분리
```

---

# 9. 외부 의존성 실패 시 기본값

문제: 서드파티 장애로 판단 로직이 흔들림.

필수 규칙

```text
외부 신호(뉴스/온체인 등) 실패 시 해당 기능 비활성화
핵심 의사결정은 내부 데이터로만 유지
```

---

# 10. 최종 “닫힘” 조건(Closure Checklist)

문서에 이 체크리스트를 그대로 넣어라.

```text
- 모든 진입/청산은 수식 + threshold + 우선순위로 정의됨
- 모든 차단 조건은 코드에서 강제됨
- LIVE는 snapshot 없이는 절대 실행 불가
- replay로 동일 결과 재현 가능
- restart 후 상태 완전 복구
- 데이터/체결 이상 시 자동 no-trade 또는 kill switch
- exposure/risk 한도 초과 불가
- dashboard에 이유(reason code) 전부 표시
- pytest 및 acceptance 전부 통과
```

# 결론

```text
이제 추가 항목의 문제가 아니라
“정의된 규칙들이 실제로 닫혀 있는지”의 문제다.
```

이 수준까지 반영되면:

```text
설계 부족으로 망하는 케이스는 거의 제거되고,
남는 변수는 시장과 execution 품질뿐이다.
```
