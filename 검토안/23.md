23차 분석 결과입니다.

판정: 불합격 유지

검토 범위:
schema instance validator, runtime artifact schema 적용 범위, JSON Schema 지원 수준

1. 자체 schema validator가 JSON Schema 일부만 지원함

파일:
`trader1/validation/schema_instance.py`

지원 확인:

* type
* const
* enum
* required
* properties
* additionalProperties false
* minLength / maxLength
* pattern
* minimum
* minItems / maxItems
* items
* 단순 `$ref`

미지원 또는 사실상 무시:

* `oneOf`
* `anyOf`
* `allOf`
* `if`
* `then`
* `not`
* `format`
* `patternProperties`
* `dependentRequired`
* `exclusiveMinimum`
* `multipleOf`

문제:
실제 schema 파일에는 미지원 키워드가 존재합니다.

확인된 schema:

* `intent_wal_event.schema.json`: anyOf
* `live_ready_candidate_writer_input.schema.json`: allOf, if, then
* `live_ready_snapshot.schema.json`: allOf, if, then
* `market_regime_adaptation_report.schema.json`: oneOf
* `official_api_verification_report.schema.json`: anyOf
* `operator_action_audit.schema.json`: allOf, if, then
* `readiness_surface.schema.json`: allOf, if, then, not
* `summary.schema.json`: oneOf

결론:
현재 schema instance validation은 통과해도 실제 JSON Schema 계약 전체를 검증한 것이 아닙니다.

2. 가장 중요한 live/readiness schema들이 조건부 검증에서 빠짐

특히 다음은 live 안전과 직접 관련됩니다.

* `live_ready_snapshot.schema.json`
* `live_ready_candidate_writer_input.schema.json`
* `readiness_surface.schema.json`
* `operator_action_audit.schema.json`

이 schema들은 `if/then/allOf/not`을 사용합니다.

문제:
자체 validator가 해당 키워드를 무시하면, live/readiness 관련 조건부 금지가 실제로 검증되지 않을 수 있습니다.

3. 테스트는 dashboard 한 개 인스턴스 중심

`tests/contract/test_schema_instance_validation.py`는 주로 dashboard 인스턴스 하나를 검증합니다.

문제:
runtime artifact 전체를 전수 검증하지 않습니다.

빠지는 것:

* summary
* heartbeat
* launcher_report
* paper_portfolio_snapshot
* startup_probe
* reconciliation_report
* live_ready_snapshot
* operator_action_audit
* official_api_verification_report
* optimizer/convergence 관련 reports

4. `runtime_schema_instance_validator`가 PASS여도 의미가 제한적임

테스트는 다음 validator 하나의 PASS를 확인합니다.

`runtime_schema_instance_validator`

하지만 자체 validator가 JSON Schema 핵심 키워드를 지원하지 않기 때문에, 이 PASS는 “부분 validator 기준 PASS”입니다.

결론:
현재 PASS를 schema contract fully satisfied로 해석하면 안 됩니다.

5. `jsonschema` 같은 표준 검증 의존성이 없음

외부 의존성을 줄인 점은 장점이지만, JSON Schema Draft 2020-12를 선언한 schema들이 있다면 표준 검증기가 필요합니다.

선택지는 둘 중 하나입니다.

* `jsonschema`를 명시 의존성으로 추가
* 자체 validator가 schema에서 쓰는 모든 키워드를 지원하도록 확장

현재 상태는 둘 다 아닙니다.

6. schema 기능과 validator 기능의 불일치가 큼

schema는 비교적 고급 JSON Schema 기능을 사용합니다.

validator는 MVP용 부분 구현입니다.

이 조합은 위험합니다. schema 작성자는 `if/then/not/oneOf`로 막았다고 생각하지만 실제 runtime validator는 해당 제약을 무시합니다.

7. 우선 패치

8. `jsonschema` 의존성 추가 또는 자체 validator 확장

9. schema에서 사용하는 모든 키워드 지원 여부를 validator가 검사

10. unsupported schema keyword 발견 시 validator FAIL

11. runtime artifact 전체 JSON 파일을 schema_id 기준으로 전수 검증

12. `system/runtime/**/*.json` 전수 schema validation 추가

13. `system/evidence/**/*.json` 중 schema_id 있는 파일 전수 검증 추가

14. live/readiness/operator 관련 schema는 조건부 검증 테스트 별도 추가

15. `oneOf/anyOf/allOf/if/then/not` negative test 추가

16. schema validator PASS 문구를 `PARTIAL_SCHEMA_VALIDATION_PASS`로 낮춤

17. Draft 2020-12 meta-schema validation 추가

결론

이번 차수에서 확인된 핵심은 schema validator가 JSON Schema 전체를 검증하지 않는다는 점입니다.

특히 live/readiness 계층에서 사용하는 조건부 제약이 실제 validator에서 무시될 수 있으므로, 현재 schema PASS는 강한 증거가 아닙니다.
