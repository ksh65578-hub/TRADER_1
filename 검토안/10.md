10차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
데이터 수집, 이벤트 파이프라인, 저장소, 운영 리소스, emergency/operator control

1. 시장 데이터 수집기가 없음

현재 확인된 public market data 구현은 다음 파일입니다.

```text
trader1/adapters/upbit/market_data.py
```

실제 내용은 STATIC_FIXTURE입니다.

부족:

* REST market metadata fetcher
* websocket collector
* ticker/trade/orderbook stream
* reconnect loop
* sequence/gap handling
* stale feed detector
* rate limit handler
* raw event writer
* canonical event converter

판정:
시장 데이터 수집 계층은 아직 구현되지 않았습니다.

2. canonical event pipeline이 없음

설계 목표상 필요한 항목:

* raw exchange event
* canonical event
* event_id / dedupe_key
* exchange / market_type / symbol / channel
* timestamp normalization
* source latency / receive latency
* raw payload hash
* canonical payload hash

현재 소스에는 ledger_event / intent_wal_event schema는 있지만, market data canonical event schema/producer는 확인되지 않습니다.

결론:
거래소 이벤트를 표준 이벤트로 정규화하는 핵심 파이프라인이 없습니다.

3. Parquet / DuckDB / Postgres 계층이 없음

검색 기준으로 실제 구현이 확인되지 않습니다.

없는 항목:

* Parquet writer
* DuckDB replay/research reader
* Postgres ledger/control store
* SQLite 대체 store
* DB migration
* schema migration
* transaction boundary
* WAL compaction
* file lock coordination

결론:
저장소 설계 표면은 거의 코드로 닫혀 있지 않습니다.

4. JSONL tick path는 validator 안의 경로 문자열 수준

mvp0_validators.py 내부에 다음 경로가 보입니다.

```text
artifact_path("data", scope, registry, "ticks.jsonl")
```

하지만 실제 tick JSONL writer/reader는 확인되지 않습니다.

결론:
데이터 산출물 경로 계약만 있고 producer/consumer가 없습니다.

5. runtime resource pressure는 실제 시스템 지표가 아님

파일:

```text
trader1/runtime/health/runtime_resource_pressure.py
```

검사 성격:

* artifact_count
* byte_count
* temp_file_count
* lock_present
* lock_age_seconds

부족:

* process memory RSS
* CPU usage
* queue backlog
* websocket lag
* event ingestion lag
* disk free space
* open file count
* DB lock state
* write latency
* event drop rate

결론:
운영 리소스 감시는 artifact directory 압력 감지에 가깝고, 실제 24/7 엔진 리소스 모니터링은 아닙니다.

6. safety_control의 정상 final_decision이 SAFE_MODE로 고정됨

파일:

```text
trader1/runtime/resource_guard/safety_control.py
```

blocker가 없어도 final_decision은 SAFE_MODE입니다.

문제:
정상 상태와 안전모드 진입 상태가 구분되지 않습니다.

권장:

* 정상: MONITOR_ONLY 또는 NO_TRADE
* blocker 있음: SAFE_MODE / KILL_SWITCH / NO_TRADE
* live 후보 판단은 별도 readiness gate로 분리

7. emergency_flatten은 dry-run 보고서일 뿐 실제 보호 동작이 없음

파일:

```text
trader1/runtime/protection/emergency_flatten.py
```

좋은 점:

* futures reduce-only path 개념 있음
* orphan position/open order review 개념 있음
* ledger/reconciliation 요구
* live permission 생성 차단

문제:

* 실제 cancel all 호출 없음
* 실제 reduce-only 주문 없음
* 실제 position close 없음
* 실제 emergency adapter 없음
* 실제 orphan position discovery 없음
* dry_run이 아니면 차단

결론:
emergency protection contract는 있으나 실제 emergency flatten 기능은 없습니다.

8. operator_control도 audit builder 중심

파일:

```text
trader1/runtime/operator_control/operator_control.py
```

좋은 점:

* action audit hash
* high-risk confirmation 요구
* manual_stop / safe_mode / reduce / exit 분류
* live permission 생성 차단

문제:

* 실제 UI/CLI confirmation flow 없음
* 실제 operator event store 없음
* 실제 kill switch latch 없음
* manual reduce/exit은 실행 불가
* order-affecting action은 reconciliation_status PASS여도 EXECUTED 금지

결론:
운영자 제어는 실제 제어 계층이 아니라 감사 레코드 생성/검증 계층입니다.

9. long-running daemon 구조가 없음

없는 항목:

* main event loop
* graceful shutdown handler
* signal handling
* restart supervisor
* reconnect supervisor
* queue worker
* bounded buffer
* backpressure policy
* metrics loop
* watchdog

결론:
“장기간 안정적으로 돌아가는 자동매매 엔진”의 런타임 골격은 아직 없습니다.

10. 이번 차수 우선 패치 목록

11. market data collector scaffold 신설

    * REST metadata
    * websocket stream
    * reconnect/backoff
    * stale detector

12. canonical_market_event schema 추가

13. raw_event_writer / canonical_event_writer 추가

14. JSONL append-only writer와 partial recovery 추가

15. data path를 exchange/market_type/mode/session_id/symbol/channel 기준으로 강제

16. feature builder가 canonical event를 읽도록 연결

17. replay runner가 canonical event stream을 재생하도록 확장

18. runtime resource monitor를 실제 process/system 지표 기반으로 확장

19. safety_control 정상 상태 final_decision 명칭 정리

20. operator_control / emergency_flatten을 report builder와 runtime actuator로 분리

10차 점수

* 데이터 수집 구현도: 1 / 10
* canonical event pipeline: 1 / 10
* 저장소 구현도: 1 / 10
* replay 입력 기반: 2 / 10
* resource monitoring: 3 / 10
* emergency protection 실제 구현: 2 / 10
* operator control 실제 구현: 2.5 / 10
* fail-closed 안전성: 8 / 10

종합: 3 / 10

결론

현재 패키지는 거래 실행 이전 단계의 안전 계약, validator, dashboard/report 스캐폴드가 중심입니다.

하지만 자동매매 시스템의 실제 핵심인 시장 데이터 수집, canonical event 처리, 저장소, replay 입력, feature 산출, 장기 실행 daemon, emergency actuator는 아직 거의 구현되지 않았습니다.
