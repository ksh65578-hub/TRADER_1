Core Objective

The system is not designed to guarantee profit.

The system is designed to maximize the probability of converging toward positive long-term expectancy by:

1. finding candidate edges
2. validating them in replay, paper, and shadow
3. eliminating negative or unstable strategies
4. allocating capital toward validated high-expectancy strategies
5. avoiding bad regimes
6. reducing risk when edge degrades
7. allowing live trading only through approved LIVE_READY snapshots

Profitability Philosophy

The target is a high-expectancy automated trading system.

Monthly 10 percent return is an aspirational target, not a guaranteed output.

The system must not chase the monthly target by:

1. increasing risk after losses
2. loosening filters
3. forcing weak trades
4. bypassing validation
5. skipping paper or shadow evidence
6. using unapproved live parameters
7. weakening drawdown controls

If monthly progress is below target, the system may only improve through:

1. better selectivity
2. better execution quality
3. better strategy selection
4. better symbol ranking
5. better capital allocation
6. better no-trade filtering

It must not increase risk merely to catch up.

Expectancy Model

Every trade candidate must be evaluated using net expectancy after all costs.

Expected value must be modeled as:

E = win_rate * average_win - loss_rate * average_loss - total_cost

Total cost includes:

1. exchange fee
2. spread cost
3. estimated slippage
4. funding cost for futures
5. partial fill penalty if applicable

A trade is not eligible if:

net_expected_edge <= 0

For high-return candidate selection, the preferred threshold is:

net_expected_edge >= 1.5 * estimated_total_cost

Gross expectancy is not sufficient.
Only net expectancy after costs is valid.

Market Regime Router

The system must classify market state before strategy selection.

Required regimes:

1. uptrend
2. range
3. downtrend
4. panic
5. quiet
6. uncertain

Example regime inputs:

1. EMA 50 and EMA 200 relationship
2. EMA slope
3. ADX
4. realized volatility
5. volatility of volatility
6. VWAP distance
7. spread percentile
8. liquidity score
9. volume expansion
10. order flow or CVD proxy if available

Regime behavior:

uptrend:
Allow pullback trend long and breakout long.

range:
Allow VWAP mean reversion only if volatility and spread are stable.

downtrend:
Block spot long.
Allow Binance futures 1x short only if short conditions pass.

panic:
No new entries.

quiet:
No new entries unless a strong breakout setup appears and all cost and liquidity checks pass.

uncertain:
No new entries.

Regime transition rule:

After regime changes, apply a confirmation window.
During the confirmation window, reduce size or block new entries.
If signals conflict across regimes, default to no-trade.

Strategy Selection

Strategies are not always active.

Each strategy is active only when:

1. current regime allows it
2. strategy-specific conditions pass
3. net expectancy after costs is positive
4. data health is valid
5. execution quality is acceptable
6. risk state allows new entries
7. no conflict with existing position state exists

If multiple strategies produce conflicting signals on the same symbol, apply:

1. safety block first
2. position state check
3. regime priority
4. strategy priority
5. net expectancy ranking
6. otherwise no-trade

Signal Grade

Every candidate signal must be graded.

Required grades:

1. no_trade
2. weak
3. normal
4. strong
5. exceptional

Weak signals must not be traded.

Normal signals may be used in paper and shadow with small sizing.

Strong signals are eligible for main paper and shadow evaluation.

Exceptional signals are eligible for larger sizing only within risk caps.

Signal score should include:

1. regime confidence
2. strategy fit
3. signal confirmation count
4. volume expansion
5. relative strength
6. trend slope
7. VWAP distance
8. breakout quality
9. liquidity quality
10. spread quality
11. net expected edge
12. historical performance of similar setup

Example grading:

score < 0.55:
no_trade

0.55 <= score < 0.65:
weak

0.65 <= score < 0.75:
normal

0.75 <= score < 0.85:
strong

score >= 0.85:
exceptional

Even exceptional signals must be blocked if:

1. net_expected_edge <= 0
2. spread is abnormal
3. slippage estimate exceeds budget
4. data is stale
5. risk cap is reached
6. position state is uncertain

Exit Logic

Every position must have exit rules at entry time.

Required exit types:

1. hard stop loss
2. signal invalidation exit
3. time stop
4. partial take profit
5. trailing stop
6. regime change exit
7. risk engine forced exit if needed

Exit priority:

1. kill switch or safety exit
2. hard stop loss
3. reconciliation or position uncertainty handling
4. signal invalidation
5. regime change exit
6. time stop
7. partial take profit
8. trailing stop
9. discretionary strategy exit if explicitly defined

Partial take profit:

At TP1, close 30 percent to 50 percent of the position.
After TP1, move stop toward breakeven if execution quality allows.
Remaining size may use trailing stop.

Invalidation examples:

Pullback invalidation:
Price fails to reclaim expected level within N candles or breaks structural support.

Breakout invalidation:
Breakout level fails to hold within N candles.

VWAP invalidation:
Range regime shifts into trend expansion against the position.

Short invalidation:
Breakdown fails and strong rebound with volume appears.

No-Trade Principle

No-trade is a valid and often optimal decision.

The system must not trade because it can trade.

It may trade only when:

1. regime is favorable
2. signal is strong enough
3. cost-adjusted expectancy is positive
4. liquidity is sufficient
5. spread is acceptable
6. slippage is acceptable
7. risk state allows entry
8. execution state is healthy
9. ledger and position state are consistent
10. data integrity is valid

If uncertain, choose no-trade.

Concurrency and Idempotency

Position state is the single source of truth.

Before submitting an order:

1. check current position
2. check pending orders
3. check risk state
4. check ledger consistency
5. check latest data health

If position state is uncertain, block new entries.

Every event must have idempotency keys:

1. event_id
2. order_id
3. trade_id
4. sequence_id where available

Duplicate events must not double count.
