있다. 지금 단계에서 빠지면 나중에 반드시 문제 되는 “운영·시장·시스템 리스크” 계층이 몇 개 더 있다. 아래는 설계도/AGENTS에 추가해야 하는 마지막 핵심 세트다.

---

# 1. 거래소 리스크(외부 의존성)

문제: API/정책/제한은 수시로 바뀐다.

필수 규칙

```text
rate_limit 예산 관리(버킷/리트라이/백오프)
listenKey/WS 재인증 자동 갱신
서버 시간 동기화(드리프트 감지 시 거래 금지)
거래소 오류코드 매핑(재시도/취소/차단 분기)
```

차단

```text
연속 API 오류 N회 → no-trade 또는 kill switch
```

---

# 2. 주문 상태 머신(Order Lifecycle)

문제: 주문은 “요청→체결”이 아니라 복잡한 상태 전이.

필수 상태

```text
NEW → PARTIALLY_FILLED → FILLED
NEW → CANCELED
NEW → REJECTED
PARTIALLY_FILLED → CANCELED
```

규칙

```text
상태 전이별 ledger 업데이트 명시
부분 체결 시 잔여 수량 처리 규칙 고정
취소/재주문 정책 명시(가격/시간 조건)
```

Acceptance

```text
어떤 상태 전이에서도 포지션/잔고 불일치가 발생하지 않는다.
```

---

# 3. 가격 정규화(Tick/Step/Lot)

문제: 거래소마다 가격/수량 단위가 다르다.

필수 구현

```text
price_tick_size 정규화
quantity_step_size 정규화
min_notional 검사
rounding rule(버림/올림) 고정
```

차단

```text
단위 미충족 주문 → 생성 자체 금지
```

---

# 4. 시간대/세션 효과(Time-of-day)

문제: 성과가 시간대에 따라 크게 달라진다.

필수 지표

```text
UTC 기준 시간대별 성과
아시아/유럽/미국 세션 성과
뉴스/이벤트 시간대 필터
```

규칙

```text
성과 열위 시간대는 진입 제한 또는 사이즈 축소
```

---

# 5. 상관관계/집중도(Correlation)

문제: 서로 다른 종목이라도 같이 움직인다.

필수 규칙

```text
상관계수 기반 노출 제한
동일 섹터/테마 과집중 제한
```

Acceptance

```text
동시 포지션이 많아도 실질 리스크는 분산되어 있다.
```

---

# 6. 모델 드리프트 감지(성능 붕괴 탐지)

문제: 잘 되던 전략이 갑자기 망가진다.

필수 감지

```text
rolling expectancy 하락
profit factor 하락
win rate 급변
slippage 증가
```

조치

```text
threshold 하회 시 자동 risk-down
추가 하락 시 해당 전략 비활성화
```

---

# 7. 재현성/결정성(Determinism)

문제: 같은 데이터로 결과가 달라지면 검증 불가.

필수 조건

```text
replay 시 동일 결과 보장
난수 사용 시 seed 고정
데이터 버전/스냅샷 해시 기록
```

Acceptance

```text
동일 입력 → 동일 출력이 보장된다.
```

---

# 8. 구성/비밀 관리(Config & Secrets)

문제: 설정과 키 관리가 깨지면 즉시 사고.

필수 규칙

```text
API key 절대 로그 출력 금지
환경변수/시크릿 스토어 사용
config schema validation
실행 전 필수 설정 검증
```

차단

```text
필수 config 누락 시 실행 금지
```

---

# 9. 감사/추적(Audit Trail)

문제: 사후 분석 불가능하면 개선도 불가능.

필수 기록

```text
모든 주문 요청/응답 원문
체결 내역
포지션 변화
리스크 차단 이벤트
```

보존

```text
시간 기준 아카이브(예: 30일 이상)
```

---

# 10. 배포/릴리스 게이트

문제: 코드가 바뀌면 성능도 바뀐다.

필수 절차

```text
replay regression 통과
paper smoke test 통과
validator 전체 통과
artifact hash 고정
```

차단

```text
게이트 미통과 시 배포 금지
```

---

# 11. 점진적 라이브 전환(Progressive Exposure)

문제: 한 번에 풀사이즈 라이브는 위험.

단계

```text
paper → shadow → micro live → scaled live
```

규칙

```text
초기 live는 최소 사이즈
일정 성과 확인 후 단계적 증액
문제 발생 시 즉시 단계 하향
```

---

# 12. 포지션 수명 관리(Position Lifecycle)

문제: 진입보다 “언제 나올지”가 더 중요.

필수 조건

```text
stop_loss
take_profit
trailing_stop
time_stop
regime_change_exit
signal_invalidation_exit
```

Acceptance

```text
모든 포지션은 최소 하나 이상의 종료 조건을 가진다.
```

---

# 13. 유동성 스트레스 테스트

문제: 평소에는 잘 되다가 유동성 고갈 시 붕괴.

필수 테스트

```text
slippage x2, x3 시뮬레이션
spread 확대 시뮬레이션
partial fill 증가 시뮬레이션
```

---

# 14. 극단 상황(Fat Tail) 대응

문제: 드물지만 치명적인 이벤트.

필수 차단

```text
가격 급변(초단기 변동성 폭증) → no-trade
호가 공백 → no-trade
데이터 갭 → no-trade
```

---

# 15. 목표-안전 우선순위 고정

가장 중요한 문장 하나를 추가해라.

```text
수익률 목표는 모든 안전 규칙보다 우선하지 않는다.
어떤 경우에도 live safety, risk limit, validation을 우회하지 않는다.
```

# 최종 요약

지금까지 추가한 것들을 한 줄로 정리하면:

```text
수익률을 높이는 구조 + 잘못된 거래를 제거하는 구조 + 시스템/시장/운영 리스크를 차단하는 구조
```

이 세 축이 모두 닫혀야 실제로 월 10%를 “노릴 수 있는 시스템”이 된다.
