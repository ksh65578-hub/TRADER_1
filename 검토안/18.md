18차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
의존성, 실행 환경 폐쇄성, import 안정성, 도구 실행 구조

1. 외부 의존성은 거의 없음

확인된 import는 대부분 표준 라이브러리입니다.

주요 import:
json, pathlib, sys, typing, hashlib, unittest, datetime, dataclasses, tempfile, subprocess, os

의미:
외부 패키지 의존성이 적은 점은 장점입니다.

문제:
그런데 `pyproject.toml`, `requirements.txt`, `pytest.ini`가 없습니다. 외부 의존성이 적더라도 실행 기준 Python 버전, 테스트 옵션, 패키지 포함/제외 기준은 닫혀 있어야 합니다.

2. Python 파일 수가 과도함

확인된 Python 파일:
341개

문제:
실제 runtime producer보다 tools/emit_* 패치 evidence 생성기가 지나치게 많습니다. 소스 프로젝트라기보다 패치 산출 도구가 누적된 작업 디렉터리에 가깝습니다.

3. tools 계층이 과도하게 비대함

`tools/emit_*_patch_evidence.py` 파일이 대량 존재합니다.

문제:

* patch-specific emitter가 계속 누적됨
* 같은 구조의 evidence writer가 반복됨
* 일부는 subprocess로 테스트/검증기를 다시 호출함
* release source에 포함될 필요가 낮은 임시 도구가 많음

결론:
도구 계층이 실제 시스템 구현보다 커져 있습니다.

4. subprocess 기반 검증 호출이 많음

확인된 패턴:

* `subprocess.run(command, cwd=ROOT, ...)`
* pytest/검증기를 emitter 내부에서 다시 실행

문제:

* 테스트 안에서 도구를 부르면 테스트가 중첩될 수 있음
* timeout 원인 가능성 증가
* 환경 의존성 증가
* 실패 원인 추적 어려움
* CI 단계 구분이 안 됨

권장:
evidence emitter는 검증을 직접 실행하지 말고, 이미 생성된 validator result를 읽어 evidence만 조립해야 합니다.

5. webbrowser import 존재

확인된 import:
`webbrowser`

의미:
대시보드나 도구에서 브라우저를 여는 경로가 있을 가능성이 있습니다.

문제:
자동검증/CI/서버/무인 실행에서는 브라우저 자동 오픈이 부적절합니다.

필요:

* 기본값은 브라우저 자동 오픈 금지
* 명시 옵션에서만 open
* headless 환경 감지

6. 테스트가 unittest 직접 실행 형태에 많이 의존

다수 테스트 파일에 다음 패턴이 있습니다.

`if __name__ == "__main__":`

이 자체는 문제는 아니지만, pytest 설정이 없는 상태에서는 테스트 실행 방식이 통일되지 않습니다.

필요:

* pytest 기준 명령어
* unittest 직접 실행은 보조 경로로 제한
* slow/release 테스트 marker 분리

7. import 계층은 외부 의존성보다 내부 결합이 문제

내부 import가 매우 많습니다.

특히:

* trader1
* tools
* tests

문제:
도구가 tests를 부르고, tests가 runtime/report builder를 부르고, emitter가 다시 subprocess로 tests를 부르는 구조가 생기면 순환적 검증 체계가 됩니다.

결론:
독립적인 검증 레이어라기보다 자기검증형 구조입니다.

8. 실행 환경 폐쇄성 부족

없음:

* Python 최소 버전 파일

결론:
다음 AI가 동일 환경에서 동일 결과를 재현하기 어렵습니다.
