2차 검토 결과입니다.

판정: 여전히 불합격

핵심 추가 발견

1. Binance futures 실행 표면이 닫혀 있지 않음

문서와 registry에는 FUTURES_USDT_M이 존재합니다.

하지만 실제 루트 런처는 둘 다 SPOT 고정입니다.

BINANCE_PAPER.py
market_type = SPOT

BINANCE_LIVE.py
market_type = SPOT

safe_launcher.py 내부도 동일합니다.

BINANCE_PAPER -> SPOT
BINANCE_LIVE -> SPOT

결론:
Binance futures 1x long/short 실행 경로가 실제로 없습니다.

2. Binance adapter 자체가 없음

현재 adapter는 Upbit만 있습니다.

trader1/adapters/upbit

없음:
trader1/adapters/binance
trader1/adapters/binance/spot
trader1/adapters/binance/futures

결론:
문서상 Binance spot/futures 계약은 크지만 구현은 아직 표면 수준입니다.

3. source bundle hygiene validator에 논리 결함 있음

source_bundle.py에서 forbidden_count를 항상 0으로 고정합니다.

즉, 금지 파일이 실제로 존재해도 forbidden_count가 증가하지 않습니다.

그런데 실제 패키지에는:

**pycache** 디렉터리 48개
.pyc 파일 340개

이 존재합니다.

결론:
검증기가 “패키지 자체에 금지 파일이 포함되어 있는지”를 제대로 실패시키지 못합니다. 현재는 “후보 번들에 포함하지 않았는가”만 보는 구조에 가깝습니다.

4. pytest 실행성이 불량함

다음 테스트 그룹도 제한 시간 내 종료되지 않았습니다.

tests/contract/test_root_launchers.py
tests/runtime/test_safe_launcher.py

원인 후보:

* build_launcher_report가 매번 전체 schema/source hash 계산
* source_bundle_manifest 생성이 전체 repo rglob 및 credential regex scan 수행
* mvp0_validators.py가 45만자 이상 단일 거대 파일
* read_only_dashboard.py가 35만자 이상 단일 거대 파일
* 테스트 import 단계에서 과도한 초기화 가능성

결론:
테스트가 “빠르게 반복 가능한 안전망” 역할을 하지 못하고 있습니다.

5. registry.yaml은 실제 YAML이 아니라 JSON임

파일명은 registry.yaml이지만 내용은 JSON입니다.

기술적으로 json.load로 읽히므로 작동은 합니다.

다만 AI 구현자나 툴링 입장에서는 혼동 위험이 있습니다.

권장:

* 그대로 유지할 경우 “registry.yaml is JSON-compatible YAML and must be parsed as JSON or strict YAML” 계약 명시
* 아니면 registry.json으로 전환

6. Upbit 구현은 상대적으로 구체적

확인된 Upbit 구현:

* paper_broker
* fee_model
* market_data fixture
* symbol_rules
* read-only account snapshot
* private stream health scaffold
* read-only reconciliation path

판정:
Upbit paper/read-only/live-review scaffold는 방향성이 맞습니다.

그러나 실제 거래소 API 연동보다는 fixture/scaffold 중심입니다.

7. live safety는 강함

확인:

* live_order_ready false
* live_order_allowed false
* can_live_trade false
* live launcher hard block
* live order gateway 차단
* order-affecting final action 차단

이 부분은 설계 철학과 부합합니다.

다만 live readiness로 가는 양성 경로는 아직 닫혀 있지 않습니다.

현재 상태 요약

좋은 점:

* fail-closed 안전 철학은 강하게 반영됨
* registry/schema/validator 구조는 존재함
* 루트 런처 4개는 존재함
* Upbit paper scaffold는 꽤 구체적임
* dashboard/readiness/evidence 표면은 많이 구축됨

중대 문제:

* Binance 실제 구현 없음
* Binance futures 실행 경로 없음
* source bundle hygiene가 실제 패키지 오염을 막지 못함
* 테스트가 제한 시간 내 안정적으로 돌지 않음
* .pyc와 __pycache__가 배포 패키지에 포함됨
* 일부 계약은 문서에만 있고 코드로 닫히지 않음

우선 수정 순서

1. 패키지에서 __pycache__와 .pyc 제거
2. source_bundle_hygiene_validator가 실제 repo 금지 파일 존재도 실패 처리하게 수정
3. BINANCE_PAPER/BINANCE_LIVE에 market_type 선택 경로 추가
4. trader1/adapters/binance/spot, trader1/adapters/binance/futures scaffold 신설
5. FUTURES_USDT_M paper simulator 최소 구현
6. Binance futures reduce-only, funding, liquidation, margin gate validator 추가
7. build_launcher_report hash 계산 캐시 또는 테스트용 fast path 추가
8. pytest 핵심 그룹이 30초 내 종료되도록 분리
9. placeholder가 readiness/live/release 표면으로 승격되지 못하게 validator 추가
10. Upbit-only 구현과 Binance surface-only 구현 상태를 current_implementation_state에 명시

2차 점수

안전성: 8.5 / 10

구현 완성도: 4 / 10
Binance 충족도: 2 / 10
