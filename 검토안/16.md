16차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
root launcher에서 runtime/dashboard artifact까지 이어지는 실제 실행 연결부

1. 루트 런처는 실제 엔진 실행기가 아님

`UPBIT_PAPER.py`는 결국 다음만 호출합니다.

```python
launcher_main("UPBIT_PAPER")
```

`launcher_main()`의 실제 동작:

* launcher report 생성
* dashboard artifact 생성
* heartbeat 1회 출력
* live 차단 상태 출력
* 종료

즉, 현재 루트 실행기는 자동매매 엔진 실행기가 아니라 “safe boot report/dashboard generator”입니다.

2. PAPER 런처도 market loop를 시작하지 않음

`launcher_main()`에서 호출되지 않는 것:

* market data collector
* paper broker loop
* strategy loop
* feature loop
* portfolio update loop
* shadow runner
* optimizer runner
* replay runner

결론:
`UPBIT_PAPER.py`를 실행해도 paper trading이 시작되지 않습니다.

3. dashboard artifact는 실제 runtime 결과가 아니라 boot-time 합성물

`build_launcher_dashboard_artifacts()`는 다음 artifact를 즉시 합성합니다.

* startup_probe
* heartbeat
* paper_portfolio_snapshot
* summary
* dashboard_shell
* dashboard_html

하지만 핵심 artifact는 기존 파일이 없으면 `None`으로 처리됩니다.

* paper_operation_gate_report
* paper_exposure_quality_report
* reconciliation_report
* restart_recovery_report
* profitability_maturity_rollup_report

즉 dashboard는 실제 trading loop 결과가 아니라 “없는 것은 누락 상태로 표시하는 boot dashboard”입니다.

4. paper portfolio는 매 실행마다 초기 스냅샷 생성

`build_initial_paper_portfolio_snapshot()`만 호출됩니다.

문제:

* 이전 paper 거래 상태를 읽지 않음
* ledger를 반영하지 않음
* 포지션을 복원하지 않음
* 손익 누적 없음

결론:
런처 실행 결과로 보는 portfolio는 실제 paper 계좌 상태가 아닙니다.

5. root launcher source hash가 `trader1/*.py`만 계산함

`source_tree_hash()`는 `trader1` 아래 Python 파일만 해시합니다.

빠지는 것:

* 루트 런처 4개
* contracts
* tests
* tools
* TRADER_1.md
* AGENTS.md

문제:
루트 실행기가 바뀌어도 source_tree_hash가 바뀌지 않습니다.

6. registry hash는 쓰지만 authority manifest와 대조하지 않음

런처는 현재 registry hash를 계산합니다.

하지만 `contracts/authority_manifest.json`의 registry hash와 일치하는지 확인하지 않습니다.

결론:
authority manifest mismatch 상태에서도 런처 report/dashboard 생성은 계속됩니다.

7. `write_json` / `write_text`는 atomic replace만 있고 fsync 없음

현재:

* tmp 파일 write
* replace

부족:

* file fsync
* directory fsync

결론:
crash/power loss 상황에서 runtime artifact 내구성은 보장되지 않습니다.

8. runtime lock은 동일 session artifact 보호용에 가까움

`.runtime_write.lock`은 유용하지만 범위가 제한적입니다.

부족:

* 전체 engine singleton lock
* exchange/market_type/mode별 중복 실행 방지
* paper/live 동시 실행 정책
* stale lock 복구 audit
* Windows power loss 후 lock recovery

9. interactive terminal에서는 continuous heartbeat로 진입 가능

`operator_pause=True`이고 heartbeat ticks가 `None`이면 Ctrl+C 전까지 반복됩니다.

즉, 터미널 실행 환경에 따라 종료되지 않는 safe monitor로 들어갈 수 있습니다.

이전 테스트 timeout의 원인 후보입니다.

10. `BINANCE_*` 런처도 동일하게 report만 생성

Binance 런처 역시:

* 실제 Binance adapter 호출 없음
* market_type은 SPOT 고정
* futures 없음
* paper loop 없음
* dashboard만 생성

종합 결론

현재 루트 실행기는 “자동매매 실행기”가 아니라 “fail-closed 상태 리포트/대시보드 생성기”입니다.

이름과 사용자 기대:

* PAPER 실행기
* LIVE 실행기

실제 기능:

* safe boot report generator
* dashboard shell generator
* live blocked proof generator

16차 점수

* root launcher 안전성: 8 / 10
* root launcher 기능 실체성: 2 / 10
* paper runtime 연결성: 1 / 10
* dashboard truth source 연결성: 3 / 10
* artifact durability: 4 / 10
* authority hash enforcement: 3 / 10

종합: 3.5 / 10

우선 패치

1. 런처 명칭/상태를 `SAFE_BOOT_ONLY`로 낮춰 표시
2. 실제 paper runtime entrypoint를 별도로 만들기
3. market loop / strategy loop / paper broker / portfolio reducer 연결
4. source_tree_hash에 루트 런처 포함
5. authority manifest registry hash 대조 추가
6. write_json/write_text에 fsync 추가
7. continuous heartbeat는 명시 옵션에서만 허용
8. dashboard에 “runtime loop not running” 명시
9. paper portfolio를 ledger 기반으로 복원
10. Binance futures 런처/market_type 경로 추가
