4차 검토 결과입니다.

판정: 불합격 유지

이번 범위:
contract / registry / schema / validator 정합성

핵심 추가 발견

1. schema_validator가 너무 얕음

현재 schema_validator는 주로 다음만 확인합니다.

* schema draft
* top-level additionalProperties false
* $id 존재 여부

문제:
실제 JSON Schema 유효성 검증, $ref 해석, required/properties 정합성, registry schema path 전체 대조까지 충분히 하지 않습니다.

즉 “스키마 파일이 존재하고 닫힌 형태인가”는 보지만, “스키마가 실제 산출물과 맞는가”는 제한적으로만 봅니다.

2. registry.yaml 이름과 내용 불일치

contracts/registry.yaml은 실제로 JSON 문법입니다.

동작은 가능하지만, 파일명과 파서 기대가 어긋납니다.

문제:
AI 구현자나 외부 툴이 YAML로 다룰 경우 혼선이 생깁니다.

권장:

* registry.json으로 바꾸거나
* “JSON-compatible YAML로만 유지한다”는 계약을 명시하고 validator로 강제

3. schema registry 매핑은 대체로 맞지만 예외가 있음

registry에는 common_defs로 등록되어 있고 실제 파일은 common.defs.schema.json입니다.

경로가 명시되어 있어 작동상 문제는 없지만, 이름 규칙이 다른 유일한 예외입니다.

문제:
자동 생성기나 AI가 “schema key = filename stem” 규칙으로 추론하면 여기서 깨질 수 있습니다.

권장:
예외를 명시하거나 key를 common_defs가 아닌 common.defs 계열로 통일해야 합니다.

4. common.defs가 전체 enum을 대표하지 않음

registry enum은 매우 많습니다.

그러나 common.defs.schema.json에 실제 enum으로 닫힌 것은 일부입니다.

확인된 주요 defs:

* exchange
* market_type
* mode
* risk_profile
* validator_status
* no_trade_reason
* live_blocker_code
* patch_class
* final_decision

문제:
나머지 optimizer/convergence/regime/failure 계열 enum은 registry에는 있지만 common.defs에 공통 투영되어 있지 않습니다.

결과:
각 schema가 자체 enum을 갖고 있거나 누락될 수 있고, drift 검증이 약해집니다.

5. closed_enum_validator가 일부 enum만 검사함

closed_enum_validator의 required 목록은 제한적입니다.

검사 대상:

* exchange
* market_type
* mode
* risk_profile
* no_trade_reason
* live_blocker_code
* patch_class

문제:
registry에 있는 수십 개 enum 중 optimizer, convergence, market regime, model drift, survival, risk scaling 관련 enum은 drift 검증 범위 밖입니다.

결론:
“registry-first enum 폐쇄성”이 전체에 적용되지 않습니다.

6. validator_registry는 크지만 실제 실행 기본값은 일부만 실행

validator_registry.json에는 implemented_validators 86개가 등록되어 있습니다.

하지만 run_validators 기본값은 MVP0_CORE_VALIDATORS 중심입니다.

문제:
MVP4 관련 validator가 많이 구현되어 있어도 기본 실행 경로에서 빠질 수 있습니다.

결과:
“등록됨”과 “기본 검증에서 실제 강제됨”이 다릅니다.

7. source_bundle_hygiene_validator가 여전히 후보 번들 중심임

검증기는 source bundle candidate의 included_files만 검사합니다.

하지만 실제 패키지에는 **pycache** / .pyc가 포함되어 있습니다.

결론:
“배포 패키지 자체 위생” validator가 별도로 필요합니다.

8. current_implementation_state가 과도하게 낙관적임

current_mvp는 MVP-4로 표시되고 completed_requirement_ids가 대량 존재합니다.

하지만 실제로는:

* Binance adapter 없음
* Binance futures 실행 경로 없음
* paper reconciliation wrapper 파손
* test 전체 실행성 불량
* fixture/report 중심 구현

문제:
current_implementation_state가 실제 구현 성숙도를 과대 표시합니다.

권장:
완료 상태를 다음처럼 분리해야 합니다.

* CONTRACT_SURFACE_DONE
* SCHEMA_SCAFFOLD_DONE
* VALIDATOR_STUB_DONE
* RUNTIME_IMPLEMENTED
* RUNTIME_TESTED
* EXCHANGE_ADAPTER_IMPLEMENTED
* LIVE_BLOCKED_ONLY

9. validator가 “스캐폴드 통과”와 “실제 구현 통과”를 충분히 구분하지 않음

여러 validator 메시지가 PASS를 반환하지만 실제 의미는 “non-live scaffold is closed”에 가깝습니다.

문제:
AI 구현자가 PASS를 실제 런타임 구현 완료로 오해할 수 있습니다.

권장:
PASS_FOR_SCAFFOLD
PASS_FOR_CONTRACT_SURFACE
PASS_FOR_RUNTIME_INSTANCE
PASS_FOR_LIVE_BLOCKED_ONLY
같은 세분 상태가 필요합니다.

10. 계약 표면은 넓지만 강제력은 불균등함

잘 닫힌 부분:

* live default false
* root launcher existence
* authority hash
* schema file presence
* 일부 runtime instance validation
* live blocked negative path

약한 부분:

* Binance 구현 강제
* futures market_type 강제
* exchange adapter completeness
* 전체 enum drift
* 전체 validator default execution
* 실제 artifact producer 존재성
* schema와 runtime output의 전면 instance validation
* 배포 패키지 금지 파일 존재 여부

4차 우선 패치 목록

1. schema_validator를 registry path 대조까지 확장
2. 모든 schema를 Draft 2020-12 meta-schema로 자체 검증
3. registry enum 전체를 common.defs 또는 schema별 enum과 대조
4. closed_enum_validator를 registry 전체 enum 대상으로 확장
5. validator_registry의 86개 validator와 VALIDATOR_FUNCTIONS, 기본 실행 세트를 상호 검증
6. current_implementation_state에 scaffold/runtime/exchange/live-blocked 상태를 분리
7. source package hygiene validator 추가
8. **pycache** / .pyc 존재 시 즉시 FAIL
9. Binance adapter completeness validator 추가
10. FUTURES_USDT_M launcher coverage validator 추가
11. PASS 메시지에 scaffold/runtime/live-readiness 의미를 분리
12. registry.yaml의 JSON-compatible YAML 정책을 명시하거나 registry.json으로 변경

4차 점수

contract surface: 7 / 10
schema 파일 구조: 7 / 10
schema 실효 검증: 4.5 / 10
registry-first 강제력: 5 / 10
validator coverage: 5.5 / 10
실제 구현 상태 반영성: 4 / 10

종합: 5.5 / 10

결론

contract/schema/validator 표면은 넓지만, 아직 “실제 구현 완료를 강하게 증명하는 검증 체계”는 아닙니다.

현재는 “문서와 스캐폴드가 많이 만들어져 있고 live는 잘 막는다” 수준입니다.
“TRADER_1 구현이 설계도대로 닫혀 간다”로 보려면 validator가 스캐폴드 통과와 실제 runtime/exchange 구현 통과를 명확히 분리해야 합니다.
