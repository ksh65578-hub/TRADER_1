있다. 지금 수준에서는 “추가하면 좋은 것”이 아니라 **반드시 넣어야 하는 운영·안전·현실 제약**이다. 빠지면 설계가 있어도 실제에서 무너진다.

아래를 설계도와 AGENTS에 명시해라.

---

# 1. 데이터 신뢰성(가장 먼저)

문제: 신호보다 **데이터 품질**이 더 큰 손실 원인이다.

필수 규칙

```text
stale_data_detect → 즉시 no-trade
timestamp 역행/중복 → 해당 구간 무효 처리
orderbook/aggTrades 불일치 → kill switch 후보
WS 재연결 후 warm-up 기간 동안 신규 진입 금지
```

Acceptance

```text
데이터 이상 상태에서 단 1건의 신규 진입도 발생하지 않는다.
```

---

# 2. 체결 품질(Execution) 분리

문제: 백테스트/페이퍼 성과와 실거래 괴리는 대부분 체결에서 발생.

필수 모델

```text
expected_fill_price
worst_case_slippage
partial_fill handling
time_to_fill
```

규칙

```text
예상 슬리피지 > 허용치 → 진입 금지
부분 체결 시 평균단가 재계산 후 리스크 재검증
미체결 timeout → 주문 취소 또는 재시도 정책 명시
```

Acceptance

```text
체결 품질 악화 시 자동으로 진입 빈도와 사이즈가 감소한다.
```

---

# 3. 비용 드리프트(수수료/펀딩/스프레드)

문제: 수수료·펀딩·스프레드는 **시간에 따라 변한다**.

필수 규칙

```text
fee_schedule runtime 반영
funding_rate 반영 (futures)
spread_percentile 기반 필터
```

차단

```text
net_expected_edge_after_all_cost <= 0 → 진입 금지
```

---

# 4. 레짐 전환 지연(오판 방지)

문제: 레짐은 항상 늦게 바뀐다.

보강 규칙

```text
regime_change_confirm_window 적용
레짐 전환 직후 N bar 동안 진입 제한
conflicting signals 발생 시 no-trade
```

Acceptance

```text
급격한 레짐 전환 구간에서 거래 빈도가 감소한다.
```

---

# 5. 과최적화 방지(Overfitting Control)

문제: 월 10% 목표는 과최적화 유혹이 매우 강함.

필수 규칙

```text
in-sample / out-of-sample 분리
walk-forward validation
min_trade_count 미달 성과는 무효
최근 성과 가중치 제한 (recency bias 억제)
```

금지

```text
단기 고수익 전략을 즉시 live 승격 금지
```

---

# 6. 자본 곡선 기반 리스크 제어(Equity-aware)

문제: 동일 sizing은 장기적으로 붕괴한다.

필수 상태

```text
equity_high
current_equity
drawdown_percent
```

규칙

```text
drawdown >= X → risk multiplier 감소
equity new high → 점진적 복원
```

---

# 7. 동시 노출(Exposure) 제한

문제: 여러 전략이 같은 방향으로 몰릴 수 있음.

필수 한도

```text
max_symbol_exposure
max_directional_exposure (long/short)
max_exchange_exposure
max_strategy_exposure
```

Acceptance

```text
동일 방향 포지션이 누적되어도 총 리스크는 상한을 넘지 않는다.
```

---

# 8. 재시작/장애 복구(Production 필수)

문제: 24/7 시스템은 반드시 죽는다.

필수 구현

```text
snapshot 기반 상태 복구
open position 재구성
미체결 주문 재동기화
ledger replay
```

Acceptance

```text
재시작 후 상태 불일치 없이 이어서 실행된다.
```

---

# 9. 이벤트 순서/중복(Idempotency)

문제: 거래소 WS는 순서 보장 안됨.

필수 키

```text
event_id
trade_id
order_id
sequence_id
```

규칙

```text
중복 이벤트 무시
순서 뒤집힘 감지 시 재동기화
```

---

# 10. 관측성(Observability)

문제: “왜 수익/손실이 났는지” 추적 못하면 개선 불가능.

필수 로그

```text
entry_reason_code
no_entry_reason_code
exit_reason_code
risk_block_reason
regime_state
signal_grade
expected_edge
realized_slippage
```

대시보드

```text
전략별 성과
레짐별 성과
시간대별 성과
비용 영향
리스크 차단 발생 횟수
```

---

# 11. 인간 인터페이스(UX)

사용자는 복잡한 조작을 원하지 않는다.

필수 버튼/상태

```text
PAPER ON
LIVE READY 여부
LIVE ON (조건 충족 시만 활성)
STOP
```

필수 표시

```text
왜 LIVE가 아직 안 되는지
지금 위험 상태가 무엇인지
거래 안 하는 이유
```

---

# 12. 목표 왜곡 방지(가장 중요)

월 10% 목표가 시스템을 망가뜨릴 수 있다.

AGENTS에 반드시 넣어라.

```text
수익률 목표는 설계 제약을 깨는 근거가 될 수 없다.
손실 한도는 절대 완화하지 않는다.
검증 단계를 건너뛰지 않는다.
과도한 거래 빈도를 강제하지 않는다.
```
