19차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
시간/타임스탬프, session_id, 결정성, hash, artifact naming, scope 충돌 가능성

1. 기본 session_id가 고정 문자열임

여러 모듈에서 기본 session_id가 하드코딩되어 있습니다.

예:
`mvp2_upbit_paper`
`mvp3_operational_paper`
`mvp3_replay`
`mvp2_restart_recovery`
`mvp4_upbit_live_review`

문제:
동일 모드 반복 실행 시 산출물이 같은 session_id 아래 덮이거나 섞일 수 있습니다.

필요:
실행마다 고유한 session_id를 생성해야 합니다.

권장 형식:
`{exchange}_{market_type}_{mode}_{YYYYMMDDTHHMMSSZ}_{short_hash}`

2. timestamp 정밀도가 초 단위임

여러 `_utc_now()` 함수가 다음처럼 microsecond를 제거합니다.

`replace(microsecond=0)`

문제:
같은 초 안에 여러 이벤트가 생성되면 timestamp만으로 순서가 보장되지 않습니다.

필요:

* monotonic sequence number
* event_index
* nanosecond 또는 microsecond timestamp
* ledger sequence

3. 여러 모듈이 자체 `_utc_now()` / `_stable_hash()`를 중복 정의함

확인:

* account_readonly.py
* market_data.py
* paper_broker.py
* private_stream.py
* reconciliation.py
* strategy_unit.py
* replay_runner.py
* dashboard 계열

문제:
시간/해시 정책이 중앙화되어 있지 않습니다. 나중에 한 곳만 바꾸면 hash format이나 timestamp format drift가 생길 수 있습니다.

필요:
`trader1/core/time.py`
`trader1/core/hash.py`
같은 공통 유틸로 통합해야 합니다.

4. artifact hash와 runtime hash가 “입력 의미”를 충분히 구분하지 않음

많은 hash가 `json.dumps(... sort_keys=True)` 기반입니다.

장점:
결정성은 어느 정도 확보됩니다.

문제:

* artifact type/version/scope salt가 없는 경우가 있음
* 같은 payload 구조가 다른 의미의 artifact에서 동일 hash가 될 수 있음
* schema version이 hash input에 항상 들어간다고 보장되지 않음

필요:
hash preimage에 다음을 강제해야 합니다.

* artifact_type
* schema_id
* schema_version
* exchange
* market_type
* mode
* session_id
* producer_version

5. paper order id 생성이 단순함

`paper_broker.py`에서 order id는 대략 다음 기반입니다.

`paper_run_id:symbol:paper`

문제:
같은 paper_run_id/symbol에서 여러 주문이 발생하면 충돌할 수 있습니다.

필요:

* intent_id
* event_sequence
* side
* timestamp
* strategy_id
* deterministic nonce

를 포함해야 합니다.

6. 같은 초 내 dashboard/artifact overwrite 가능성

런처와 dashboard가 고정 파일명으로 산출물을 씁니다.

예상 구조:

* dashboard_shell.json
* summary.json
* heartbeat.json
* paper_portfolio_snapshot.json

문제:
실행 이력 보존과 최신 상태 파일이 분리되어 있지 않으면 이전 실행 증거가 덮일 수 있습니다.

필요:

* latest pointer
* immutable run artifact
* session-scoped artifact directory

7. session scope와 artifact path 강제력이 부족함

검증 로직은 report 내부의 `exchange/market_type/mode/session_id`를 비교하지만, 파일 경로 자체가 그 scope를 강제하는지는 약합니다.

필요:
`artifacts/{exchange}/{market_type}/{mode}/{session_id}/...`

그리고 validator가 파일 경로와 내부 scope 불일치를 실패 처리해야 합니다.

8. evidence accumulator가 session_id를 aggregate 문자열로 합성함

`evidence_accumulator.py`에서 다음 패턴이 보입니다.

`{paper_session_id}_aggregate`
`{paper_session_id}_shadow_aggregate`

문제:
집계 artifact와 원본 세션 artifact가 같은 ID 체계 안에 섞입니다.

필요:

* aggregation_id 별도 필드
* source_session_ids 배열
* derived_from_artifact_hashes
* aggregation_window_start/end

9. 시간 기반 freshness 검사는 clock skew에 취약함

dashboard의 freshness 계산은 현재 시각과 generated_at 차이 기반입니다.

문제:

* 시스템 시간이 바뀌면 age_seconds 왜곡
* future timestamp 처리 필요
* monotonic clock 기반 runtime freshness와 wall-clock display time 분리 필요

10. 이번 차수 우선 패치 목록

11. session_id 생성기를 중앙화

12. 고정 session_id 기본값 제거

13. timestamp에 microsecond 유지

14. ledger/event sequence 필드 추가

15. `_utc_now`, `_stable_hash` 공통 유틸로 통합

16. hash preimage contract 추가

17. order id/client id 생성기 중앙화

18. artifact path를 exchange/market_type/mode/session_id로 강제

19. latest artifact와 immutable artifact 분리

20. aggregation_id와 source_session_ids 분리

21. freshness 검사에 future timestamp / clock skew 방어 추가

19차 점수

session_id 안전성: 2 / 10
timestamp 정밀성: 4 / 10
hash 결정성: 6 / 10
hash 의미 분리: 3 / 10
artifact overwrite 방지: 3 / 10
scope/path 강제력: 4 / 10

종합: 3.5 / 10

결론

현재 패키지는 단일 dry-run report 생성에는 큰 문제가 없어 보이지만, 반복 실행·장기 실행·다중 세션·paper/live 병렬 운영으로 확장하면 session_id, artifact path, timestamp, hash 충돌 문제가 발생할 가능성이 큽니다. 고정 session_id와 중앙화되지 않은 시간/해시 유틸은 반드시 정리해야 합니다.
