17차 검토 결과입니다.

판정: 불합격 유지

검토 범위:
Windows/로컬 24시간 운영성, 파일 잠금, graceful shutdown, 로그/보존 정책, 사용자 운영 UX

1. graceful shutdown 구조가 없음

검색 기준으로 다음 처리가 확인되지 않았습니다.

`SIGINT`
`SIGTERM`
`CTRL_C_EVENT`
`atexit`
`graceful_shutdown`
`shutdown_handler`

문제:
실제 장기 실행 paper/live 엔진이 붙으면 종료 시 다음을 보장하지 못합니다.

* 신규 진입 중단
* open intent flush
* ledger/WAL flush
* dashboard final snapshot write
* lock 해제
* partially written artifact 복구
* 마지막 heartbeat 갱신

현재는 report generator 수준이라 덜 드러나지만, 실제 엔진으로 확장하면 필수 결함입니다.

2. Windows 전용 종료/잠금 대응이 없음

현재 lock은 `os.open(... O_CREAT | O_EXCL ...)` 기반입니다.

장점:

* 단일 파일 writer lock으로는 단순하고 유효함

부족:

* Windows CTRL_C_EVENT 처리 없음
* Windows power loss 후 lock recovery audit 부족
* msvcrt/portalocker 등 플랫폼별 advisory lock 없음
* 동일 exchange/market_type/mode engine singleton 보장 없음
* paper/live 동시 실행 정책 없음

결론:
Windows 로컬 24/7 운영 목표 기준으로는 아직 약합니다.

3. artifact write는 atomic replace만 있고 durability가 부족함

`safe_launcher.py`의 `write_json`, `write_text`는 다음 구조입니다.

* tmp write
* replace
* tmp cleanup

부족:

* file flush
* `os.fsync(file)`
* parent directory fsync
* write 실패 시 partial artifact quarantine
* corrupted JSON artifact 복구

결론:
프로세스 종료 정도에는 어느 정도 안전하지만, 전원 장애/OS 크래시까지 고려한 crash-safe write는 아닙니다.

4. 로그 시스템이 없음

확인되지 않음:

* central logger
* rotating log
* JSONL operation log
* error log
* redacted exception log
* per-exchange log path
* per-mode log path
* retention purge

현재 대부분은 JSON report와 dashboard artifact 중심입니다.

문제:
장기 운영 시 원인 추적이 어렵습니다.

필요:

* `logs/{exchange}/{market_type}/{mode}/YYYY-MM-DD.jsonl`
* secret redaction
* exception sanitizer
* retention 30일 기본값
* startup/shutdown/recovery event logging
* order/reconciliation/no-trade reason logging

5. 보존/정리 정책이 없음

검색상 purge/retention/cleanup 구조가 실질적으로 없습니다.

문제:
runtime artifacts, evidence, dashboard html/json이 계속 누적될 수 있습니다.

이미 현재 패키지에도 `system/evidence` 산출물이 대량 포함되어 있습니다.

필요:

* runtime artifact retention
* dashboard snapshot retention
* logs retention
* temp file cleanup
* old evidence archive 정책
* zip/source package에서 runtime artifact 제외

6. runtime stability history는 실제 안정성 측정이 아님

`stability_history.py`는 dashboard shell의 metric status를 history로 누적합니다.

문제:
실제 운영 안정성 지표가 아닙니다.

부족:

* process uptime
* restart count
* event loop lag
* websocket reconnect count
* dropped events
* DB/write latency
* memory RSS
* queue backlog
* API error count
* recovery success/failure

결론:
현재는 dashboard 표시 상태의 history일 뿐, 장기 운영 안정성 계측은 아닙니다.

7. runtime resource pressure도 artifact directory 중심

`runtime_resource_pressure.py`는 주로 다음만 봅니다.

* artifact_count
* byte_count
* temp_file_count
* lock_present
* lock_age_seconds

부족:

* CPU
* memory
* disk free
* file descriptor count
* queue length
* websocket lag
* event ingestion lag
* DB lock contention
* write latency

결론:
장기 운영 resource guard로는 부족합니다.

8. 사용자 운영 UX는 “보기” 중심이고 “제어”가 약함

현재 dashboard/launcher는 상태 표시와 live 차단 메시지는 많습니다.

하지만 실제 사용자 운영 플로우에 필요한 기능이 없습니다.

부족:

* paper engine start/stop 상태
* shadow running 여부
* data collector running 여부
* last market event time
* strategy loop running 여부
* safe shutdown button/command
* pause new entries
* close paper session
* export session report
* clear stale lock guidance
* recovery required guidance

결론:
사용자 친화적 dashboard의 골격은 있지만, 실제 운영 제어 UX는 미구현입니다.

9. “장기간 안정적으로 돌아감” 기준에서 현재 상태

현재 충족:

* live 기본 차단
* runtime write lock 일부 존재
* artifact pressure 감지 일부 존재
* dashboard shell 존재
* heartbeat 존재

현재 미충족:

* daemon loop
* graceful shutdown
* durable ledger/WAL
* rotating logs
* retention cleanup
* real resource monitoring
* recovery procedure
* market feed liveness
* paper session continuity
